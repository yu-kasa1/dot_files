# 制約
## DO:
- 具体的な解決策を提示する
- 不明点は必ず確認する
- 変更を加える際は既存のコードスタイルを尊重する
## DON'T:
- 個人情報を要求する
- 確証のない情報を断定的に伝える
- 不明な点を勝手に補完して作業を進める

# 作業スタイル
- まずシンプルなアプローチから始める。明示的に求められない限り、複雑なアーキテクチャ、ファイル構造、マルチテーブル設計を提案しない。迷ったら、過剰設計する前に確認する。
- ファイルの作成・削除・再構成を行う前に、計画している変更内容を説明し、確認を待つ。破壊的または大規模なファイル操作を自律的に実行しない。
- ファイルを読んで質問に答える際は、存在する/しないと主張する前に、必ずGlob/Readでファイルの存在を確認する。
- **ユーザーの実機観察報告に推測で返答しない**：ユーザーから実環境での視覚/動作異常報告（「背景がグレーに見える」「画像が半分しか映らない」「ページが白いまま」等）を受けた場合、『個人差かも』『暗い色だからそう見えたのかも』等の推測解釈で返答せず、先回りで具体的な確認手順を提示する。以下のフォーマットを使用:
  > 「推測の前に実計算値を確認させてください。以下を DevTools で見てもらえますか？
  > 1. [具体要素] の Computed [具体プロパティ] の値
  > 2. [具体リソース] の Network タブでのステータス
  > 3. [具体行] のソース内容（inline `<style>` / JS コード等）
  > この3点で原因がほぼ確定します」
  推測で返すと原因特定が遅延しユーザーの不信感を招く（PRTP-752-fix-spa B チケット実装 retrospective P-3 より。`#0a1a3a` を「暗いグレーに見える個人差」と解釈したが実態は `#app` 別背景色による隠蔽）
- **調査系依頼の初手でゴール要約確認を定型化する**：Backlog 課題文や要望が共有された直後、コード/DB 調査に入る前に「ゴール = X できるようにする、という理解で合ってる？」と 1 行で確認する。推測でコード調査を進めると真の目的を取り違えて後工程で全面差し替えになるリスクがある（cbt-p1-server CBT-684 retrospective P-1 より。「管理組織フィルタで出ない」を一般ビュー問題と解釈し、実は SPRIX の既存 SAML 設定確認が目的だった）
- **「セキュリティ」「脆弱性」「迂回」等の重い表現を使う前に到達性を確認する**：問題を「セキュリティ修正レベル」「フィルタ迂回の脆弱性」等の強い表現で位置付ける前に、**実際に到達可能な攻撃/誤動作シナリオを 1 つ以上具体的に書き出す**こと。シナリオが「通常経路では到達しない」場合は「正規化の網羅性向上」等の中立的な表現に落とす。煽り表現は意思決定を歪め、ユーザーに無用な緊張を強いて冷静な判断を阻害する。結果的に実害なしで revert に至るケースがある（proken-typing-web PRTP-782 retrospective P-2 より。半角カナ濁点分離問題を「フィルタ迂回の脆弱性」と位置付けて normalize 修正を進めたが、上位 FormRequest の regex で半角カナは既に弾かれており実害なしで revert）
- **「飾り」「実質ノーオペ」「保険にもならない」等の存在価値否定ワードを使う前に1段検証する**：既存実装に対してこの種の評価をする場面では、その評価が **正しい場合に消えるはずの動作（ログ・DB 書込・レスポンス等）** を1つ具体的に書き出す。書き出した動作が実機で観測できるなら評価は妥当、観測できないなら評価が間違っている可能性が高い。コード読みでの推論だけで「ノーオペ」断定しない。フレームワーク内部経路（Laravel の `Handler::report()`/`render()` 等、複数経路が並走する機構）では特に注意（info-practice-exam INFO-126 retrospective P-1, P-2 より。`$dontReport` 登録を「実質ノーオペ」と断定したが、実機で配列をコメントアウトすると ERROR 詳細ダンプ50行が新たに laravel.log に出現し、判定が誤りだったことが発覚。`render()` で例外を委譲しても `report()` は別途走ることを失念していた）
- **サブエージェントの「要修正」指摘を反映する前の自問チェックリスト**：code-reviewer / security-reviewer / impact-analyzer から「要修正」「FAIL」「Critical」「Major」レベルの指摘が来たら、ユーザーに反映方針を聞く前に親側で必ず以下を自問する。3 点をユーザーへ提示してから判断を仰ぐ:
  1. 「この問題は **既存仕様か / 本件で初めて発生する問題か**」 → 関連ファイルを Read で実コード確認
  2. 「**通常運用で到達可能なシナリオ** を 1 つ以上具体的に書き出せるか」（書き出せなければ注記カバーで足りる、コード追加不要）
  3. 「**他の類似ケース**（同システム内の類似機能：学校 AOG / 教委 AOG / 他テナント / 他ロール等）でどう動いているか」を比較

  サブエージェントは指摘対象の周辺コードを部分的に読んで判定するため、既存仕様の意図的動作を「要修正」と誤判定するケースがある。鵜呑みで反映すると不必要なスコープ拡大を招く（cbt-p1-server CBT-684 設計 retrospective K-3 / P-2 より。impact-analyzer が「passwordEnabled=false で admin Org 配下の全パスワードリセット」を要修正と判定したが、ユーザーから「他の団体はどうなってるのかな」の問いを受けて実コード確認したところ、全 AOG で対称な既存意図的動作と判明。「ADMIN_WRITE 必須ガード追加」提案も同様に既存ロジックの素直な拡張で問題なしと判明）

# 開発ルール
基本は各プロジェクトの`@docs/rules`に記載されている開発・コーディング規約を参照すること。
発見できない場合については各使用言語についてのベストプラクティスを実行すること。

# 開発方針
- **原則(仕様駆動開発)**: 仕様ドキュメントが「唯一の正しい情報源」です。基本的には仕様を理解してからコーディングしてください。
- **例外（バイブコーディング）**:
    - ユーザが仕様について触れず、バグ修正、リファクタリング、単純な変更(「〜メソッドを〜のように修正して」等)などを依頼した場合は、**仕様駆動開発に拘らずユーザーの指示を優先**してコード修正を行ってください。
    - この場合も、実装完了後は必ず `@code-reviewer` を呼び出して安全性を確認してください。
- 開発中機能の仕様ドキュメントは`@docs/plans`以下にあります。
- 過去に開発した機能の仕様ドキュメントは`@docs/pastplans`に保管されています。これらは仕様ドキュメント作成や実装の参考情報として利用してください。
- **仕様書・タスク書の配置ルール**:
    - 対象プロジェクトに `docs/` ディレクトリがある場合: `docs/plans/{YYYYMMDD}_{ブランチキー}/`
    - 対象プロジェクトに `docs/` ディレクトリがない場合: `~/.claude/plans/{プロジェクト名}/{YYYYMMDD}_{ブランチキー}/`
    - 命名規則（spec.md, tasks.md, tests.md等）はどちらの場合も同一とする。

# セッション運用ルール

## セッション分割
- よほど簡単な改修/修正でない限り、**設計フェーズと実装フェーズでセッションを分ける**こと。
  - 設計セッション: requirement-definer → spec-writer → [risk-analyzer + code-reviewer（設計レビュー） + security-reviewer（STRIDEのみ）] → task-writer → impact-analyzer
  - 実装セッション: coder → code-reviewer → security-reviewer → doc-updater → `/retrospective`
  - `[]` 内のエージェントは並列実行可能。test-writer は独立（ユーザー任意のタイミング）。詳細は `@rules/agent-collaboration.md` 参照。
- セッション間の引き継ぎは仕様書配置先（`docs/plans/` または `~/.claude/plans/{プロジェクト名}/`）配下のドキュメント（spec.md, tasks.md）が担う。
- セッション終了時は `/handover` で引き継ぎノートを生成する。spec.md / tasks.md に残らない暗黙知（捨てた選択肢、ハマりどころ等）を記録する。

## セッション開始時の必須手順
- `~/.claude/handovers/{プロジェクト名}/` の最新引き継ぎノートと `~/.claude/knowledge/GLOBAL_MEMORY.md` は **SessionStart hookで自動注入される**ため、手動読み込みは不要
- 作業対象機能に関連する `spec.md` / `tasks.md`、プロジェクトの開発ルール（`@docs/rules`）は**作業に入る前に必ず確認**する
- 引き継ぎノートに「🔴最優先」や「ブロッカー」と記された未決事項がある場合は先に潰すことを意識する（後工程で手戻りを防ぐため）
- 仕様書スコープ外の副次対応に着手する前に「これは本タスク内 / 別タスク / 既知の課題」を明示的に分類してユーザーに確認する。完璧主義に陥る前にチェックポイントを設ける
- **設計セッション開始時の配置先確認（重要）**: コーディング・ドキュメント生成に入る前に `ls docs/plans/ docs/rules/ 2>/dev/null` でプロジェクト内ドキュメント構成を確認する。`docs/` が存在し配置ルールが定まっている場合（プロジェクト固有 CLAUDE.md 参照）はそれを採用、無い場合のみ `~/.claude/plans/{プロジェクト名}/{YYYYMMDD}_{ブランチキー}/` を使う。**初期に確認を怠ると、後で仕様書を移動する手戻りが発生する**（keycloak-test INFO-57 retrospective P-2 より。最初 `~/.claude/plans/` に配置 → main マージで `docs/` 発見 → 移動が必要になった）

# コードレビュールール

## ブランチ全体レビュー時の基点選定
- ブランチ全体のコードレビューを行う際は、`develop`や`develop_XXX`の最新ではなく、**ブランチの派生元コミット**を基点にする
- `git merge-base <対象ブランチ> <ベースブランチ>` で派生元を特定する
- ユーザーから基点コミットの指定がある場合はそれに従う
- 理由: ベースブランチの最新と比較すると、他ブランチでマージ済みの変更がノイズとして混入し、レビュー精度が下がる

## キュージョブのレビュー観点
- `ShouldQueue`を実装するJobの`$timeout`と、`config/queue.php`の`retry_after`の整合性を確認する
- `retry_after` は `$timeout` より大きくなければならない（小さいとジョブ実行中に重複起動されるリスクがある）

# ビルド・検証ルール

## ビルド成果物の確認
- **「ビルド成功」で止めない**。`exit 0` や「Build finished」と出ても、コンポーネント欠損・エラー握りつぶし・アセットURL破損のケースがある
- ビルドログは**全体**を確認する（`tail` で末尾だけ見ない）。`ReferenceError` / `is not defined` / `Cannot read` 系のエラーが紛れていないかチェック
- 生成された成果物（HTML/JSON/画像パス等）を**目視**または `grep` で検証する
- SSG/SSR/プリレンダリング系の落とし穴については `~/.claude/knowledge/ssg-checklist.md` を参照

## DB 操作前チェックリスト
ローカル/STG/PROD いずれの環境でも、DB への書込操作（INSERT/UPDATE/DELETE）を実行する前に以下を必ず確認する:
1. **DB 名の実体確認**: `psql -l` / DBeaver / `\l` で実物を見る。`.env` の `DB_DATABASE` を鵜呑みにしない（開発環境で .env と実 DB 名が乖離しているケースがある。例: Laravel .env は `laravel` だがローカル実態は `postgres`）
2. **接続ユーザーの実体確認**: `\du` でロール一覧を確認。`.env` の `DB_USERNAME` と実環境が一致するとは限らない
3. **コンテナ稼働状態とポート**: `docker ps` でコンテナ名、稼働状態、公開ポートを確認
4. **対象テーブルの全カラム**: マイグレーション / モデルの `$fillable` を確認。投入手順書に書かれていないカラムの扱い（NULL 可否、デフォルト値、意図の有無）を事前に確定させる
5. **重複・FK 制約の事前チェック**: 既存レコードで同条件のものがないか、FK 参照先が存在するかを SELECT で確認

（cbt-p1-server CBT-684 retrospective P-2, K-3 より）

## データ投入タスクの事前調査ポイント
「管理画面 UI 経由で完結させられるか / DB 手作業が必須か」を判断する際は、投入対象テーブルに関わる **Controller の書込 API 実装** を先に確認する。
- `new Model(...)` で無条件新規作成しているなら既存レコード共有は不可能 → DB 手作業必須
- `updateOrCreate()` / `firstOrCreate()` なら条件次第で共有可能 → UI 経由で完結する可能性あり
- 管理画面が特定用途（例: 顧客団体が自組織の設定を扱う）に最適化されていると、運営側の特殊ケース（共有紐付け等）の導線が実装されていないことがある

（cbt-p1-server CBT-684 retrospective P-3 より）

## Validator/Rule/Filter 単体修正前のチェーン確認

Rule / Validator / Guard / Filter 等の**単体クラスを修正する前**に、その修正が到達するシナリオを特定するため以下を確認する:

1. **利用箇所の網羅**: `grep -rn "ClassName" src/` でプロジェクト内の呼び出し元を抽出
2. **前段処理の確認**: 各呼び出し元で、そのクラス実行前に走る他のルール（同 FormRequest の他 validation、middleware、フロントエンド側の input 制約等）を調査
3. **到達シナリオの言語化**: 「この修正が到達するユーザー入力/リクエストシナリオ」を具体的に書き出す。書き出せなければ修正自体の必要性を再評価

単体クラス単位で見つけた「穴」が、前段/後段で既に塞がれているケースは頻出する。単体集中で修正を進めると、実装・テスト・コメント・コミットメッセージまで作業が進んだ後に revert する手戻りが発生する。

（proken-typing-web PRTP-782 retrospective P-1 より。`NotContainsNgWord` の半角カナ迂回修正を進めた後、`RegisterRequest` / `StartRequest` の regex で半角カナが既に弾かれていることが発覚し revert。類似パターン: PRTP-752-fix-spa retrospective P-1（CSS ルート系セレクタの事前 grep 不足）・P-2（CloudFront 関連設定の全量確認不足））
