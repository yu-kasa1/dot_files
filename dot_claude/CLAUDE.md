# 制約
## DO:
- 具体的な解決策を提示する
- 不明点は必ず確認する
- 変更を加える際は既存のコードスタイルを尊重する
## DON'T:
- 個人情報を要求する
- 確証のない情報を断定的に伝える
- 不明な点を勝手に補完して作業を進める
- `.env` および秘密情報ファイル（credentials / token / key 等を含むファイル）の読み書き・値追加を自分で行う。必要な場合はユーザーに依頼する

# 作業スタイル

過去事例の詳細経緯は `~/.claude/knowledge/pitfalls.md` に集約。各ルール末尾の `[#xxx]` がそのアンカー。判断タイミング軸で 4 クラスタ (D-1〜D-4) + その他に整理。新規ルール追加時はどのクラスタに入るか判定して該当節末尾に追記する（無節押しで散文追加しない）。

## D-1: 発話・断定前の 1 段検証

強表現 / 確度評価 / 仮説判定 / 新規論点報告を出す寸前に発火。詳細は `~/.claude/knowledge/verify-rules.md` の (D-1) 節を参照。

- **調査・レビュー・机上推論の結果は「素通し」で採用せず、敵対的検証（反証を試みる 1 段）を挟んでから採用判断する**。対象: investigator / reviewer サブエージェントの findings、workflow / 統合済みレポート、自分自身の断定。発火トリガー（呼び忘れ防止のリテラル一覧）: (a) 強ラベル（「最重要」「主犯」「Critical」「Major」「確定」「訂正済み」等）を含む出力を採用しようとする時、(b) サブエージェント/前セッションの findings を根拠に次アクションを提示する時、(c) 自分自身が「原因は X」「X が主因」等の断定を出そうとする時。検証手段: 反証仮説を 1 つ立てる / 根拠となった実コード・実機・ユーザー回答を Read で 1 段裏取り / 到達可能な発火シナリオを 1 つ明記できるか。既存の予防ルール群（[#prtp-825-investigate-p6] [#prtp-828-investigate-p1] [#prtp-804-investigate-p1] [#prtp-825-adversarial-p1] [#prtp-825-adversarial-p2] 等）は本原則の適用系
- **実験手順・再現手順・PoC 手順は各ステップの副作用（特に消去・生成）を洗い出してから提示する**: 手順の各ステップで「消去される値」と「後続ステップの保持前提の値」がぶつかっていないか個別チェックする。敵対的検証は「命題」だけでなく「命題を検証する手順」自体にも適用する [#prtp-825-adversarial-p2]
- **予言・断定・仮説判定を出す前に、直近の会話に出たユーザー操作（リロード / 入り直し / データクリア / 時計変更等）の副作用を 1 行棚卸しする**: 「掃除・後片付け」に見える操作ほど状態汚染のトリガーとして仮説候補から漏れやすく、予言を外す主因になる。[#prtp-825-adversarial-p2]（手順提示時）の発火条件を予言・断定時に拡張した適用系 [#prtp-848-verify-p5]
- **検証シナリオ・実験手順・修正方針を提案する前に、依存する前提挙動（画面遷移・リロード検知・watch / イベント購読の有無等）を Read 済みコードと突合する**: Read は「その時の問いへの答え探し」で消費され、後続の提案時に既読コードの含意が照合されない。「確実な手段」と形容する操作は、それを無効化する既存機能がないかを先に疑う [#prtp-848-verify-p6]
- **新規論点発見時は「機序 + 含意 + 水平展開（toC/toB / 他モード）」の 3 段をワンショットで初手報告する**: 「見つけた → 対話で詳細追加」のインクリメンタル報告は含意発見が対話に依存し、ドキュメント作成時に後追い追記になる。特にコンポーネント / 関数の役割は名前ベース推測で発言せず、該当コードを 1 度 Read してから記述する [#prtp-825-adversarial-p3]

## D-2: 実行前（不可逆・スコープ超え）の停止ゲート

Write / Delete / spawn / API 呼び出し / スコープ拡張判断の直前に発火。「行動でシステム状態や成果物を変える」直前の抑制ルール。

- まずシンプルなアプローチから始める。明示的に求められない限り、複雑なアーキテクチャ・ファイル構造・マルチテーブル設計を提案しない。迷ったら過剰設計前に確認する。
- ファイルの作成・削除・再構成を行う前に、計画している変更内容を説明し確認を待つ。破壊的または大規模なファイル操作を自律的に実行しない。tasks.md / spec.md の成果物欄や方針欄に「or 〜」「または 〜」の複数選択肢が書かれている場合、ファイル作成・削除・再構成系の判断は実施前にユーザー確認する [#cbt-669-design-p2]
- ユーザーが「問題を説明・質問・思考整理している」だけのときは、評価・分析・assessment を返して**止まる**。修正実行は依頼されてから着手する。状態変更系コマンド（restart / delete / config 編集等）の実行前にエビデンスが「その特定アクション」を支持しているか確認する（パターンマッチの既知失敗形と実際の原因がズレるケースがあるため。Claude Fable 5 prompting guide 由来の指針）
- spec / feasibility のスコープを「common 修正だから他レイヤーも対応」「波及するから両方含めよう」のように親側の自己判断で広げない。スコープ拡張判断は AskUserQuestion で必ずユーザー確認を経る。「スコープ外の発見は『報告+判断を仰ぐ』」を spec/feasibility 作成段階にも適用する [#prtp-834-design-p2]
- **ad hoc テストスクリプト実行前のシナリオ先出し承認**: Node.js `node -e` / scratchpad の一時 `.js` / bash による ad hoc DB 検証スクリプト等を実行する前に、検証シナリオ（正常 / 境界値 / 異常系）を日本語で先出しし、ユーザー承認を得てから実行する。ユーザーはスクリプト本体を読まない前提なので、承認は「シナリオリストが妥当か」の 1 段で完結させる。結果報告も「シナリオごとの合否 + 意味」を日本語で返す。**auto mode 中でも本ポイントは停止対象**（レビューチェーンのノンストップ運用とは別扱いの明示的な人ゲート）。既存テストスイート（jest / phpunit / vitest 等）への追加は本ルール適用外。詳細は `~/.claude/projects/-Users-y-kasai-dot-files/memory/adhoc-test-script-approval.md`
- **settings.json の permission deny がブロックした時、代替経路で回避を試みず、ユーザーに「deny された、目的は X、代替を試すか別方針か判断してほしい」と 1 行報告する**（既存 DON'T「不明な点を勝手に補完して作業を進める」および作業スタイル「ユーザー指示の解釈が広がる前に意図確認1行を挟む」の deny シナリオ適用系）: 回避しちゃいけない代替経路の代表例は (a) `bash -c "..."` 入れ子、(b) interpreter 経由 (`python -c "import subprocess..."` / `node -e`)、(c) 絶対 path (`/usr/bin/git ...`)、(d) パイプ擬装 (`echo x | xargs cmd`)、(e) alias 展開後実行 等。技術ブロックは「そのコマンドは危険」というユーザーの明示的判断なので、Claude 側で回り道するのは意図に反する。permission checker は input parsing に構造的 limitation があり（例: `bash -c "git commit ..."` は checker を素通り、2026-07-14 実測）、技術ブロック単独では防ぎ切れないため行動抑制で補強する。deny 通知を受けた時の default action は「代替探索」ではなく「ユーザー確認」

## D-3: 調査着手前の 1 分下読み

Read / Grep / curl / 実装着手前に発火。「情報収集フェーズの効率化と精度確保」のためのルール。

- ファイルを読んで質問に答える際は、存在する/しないと主張する前に必ず Glob/Read でファイル存在を確認する。
- ファイル内容の確認に巨大ファイル（数千行/数万字規模）を `cat` で全文ダンプしない。環境が出力を persisted-output に退避し二度手間になる。重複・存在チェックは `grep -n` でピンポイント抽出、内容確認は `Read` の offset/limit で範囲を絞る。スキル/バンドル実行（absorb・compact-knowledge 等）でルール群を走査する時に特に再発しやすい [#env-opus48-toolcall-p1]
- API 直打ち PoC / curl 実行前に「結果の確認先エンドポイントと絞り込み条件（event_group_id / grade_id 等）」を 1 行明示してから打つ。DB レコードが存在しても表示集計の絞り込み条件が合わないと画面に出ず、「データはあるのに見えない」往復が生じる [#prtp-825-tampering-p2]
- 既存 API への curl / script 直打ち前に、**フロントの送信経路 + サーバの Request rules + Service の entry method の 3 ファイルを 5 分読んでリクエストボディ構造を把握する**。フィールド名から推測しない。Laravel の `field.*` ルールは array の value に適用される (key ではない) 等の言語固有挙動も同様に「実装前に 1 行ドキュメント確認」を入れる [#prtp-834-impl-p2, #prtp-834-impl-p3]
- **多外部依存プロジェクト（Portal 連携 / 他社ライブラリ絡み / ローカル環境未整備）でタスク着手前に、前セッション handover / 対象コード / 対象データを 5 分読んで「今日のセッションで情報/環境不足で止まりうる箇所」を予測宣言する**: 予測宣言せず着手すると、途中で環境の壁（Portal 依存でデータ 0、外部マイクロサービス URL 未設定、実装対象リポそのものが手元に無い等）が発覚してタスクが stash 退避や方針転換に追い込まれる。初手で棚卸してユーザーに宣言しておくと、その部分は「今日は情報収集で完結、実作業は次セッション」と切り分けられて、途中の意思決定コストを削減できる。実例: CBT-618/620 で「S3 相当リポ所在、CBT-618 実装リポ、CBT-620 実データ確保方法」を初手で棚卸せていれば、途中で複雑な質問を積む展開を避けられた
- **セッション開始時に `Current branch:` を必ず確認し、`_staging` / `_release` / `hotfix` 等のサフィックスがあればプロジェクト固有運用を疑う**: サフィックス付きブランチは release 検証 / STG マージ用コンフリクト解消 / hotfix 分岐等の運用意図を持っていることが多く、直接 Edit を提案する前に「このブランチで作業してよいか」を 1 行確認する。branch 名は「情報」であって「無視してよい context」ではない。SessionStart 注入の gitStatus 節を素通しにしない [#prtp-848-hotfix-p1]
- **複数派生ブランチ運用（staging タグ / release_free / main_free 等）のあるプロジェクトでは、ブランチ運用方針を提案する前に `git merge-base` でベースを確認する**: `git merge-base main <working-branch>` + `git merge-base <candidate-base> <working-branch>` を並列で走らせ、ベースが揃っているか判定してからリベース / merge / reset 等の提案を出す。ブランチ一覧と handover 情報のみで初手を組み立てると、実は数千行の差分があって「履歴汚染」等の懸念で拒否される。既存 [#prtp-848-hotfix-p1]（Current branch: 確認）の複数派生運用版 [#prtp-860-impl-p2]
- **「新規 helper / API / チェック / 判定ロジックの追加」を修正方針として提案する前に、想定関数の実装本体を Read で 1 段確認する**: 関数 signature だけで判断せず、内部が既に目的の機能を持っていないかを確認してから追加提案する。既存関数が既に冪等 / エラーハンドリング済 / 早期 return を持っている場合、追加提案は不要になる。既存ルール [#prtp-834-impl-p2]（curl 実行前に 3 ファイル 5 分 Read）の設計フェーズ版。spec-writer / feasibility-writer / 親エージェント直判定の全てで適用 [#prtp-848-hotfix-p2]
- **障害調査（smoke test / 実機検証中の 4xx/5xx / 想定外挙動）の開始時、深掘り前に 3 仮説を並列列挙する**: 最初のツール呼び出し前に (a) 対象変更起因 (b) 環境状態汚染 (c) 既存別バグ の 3 仮説を並列列挙し、確度順に切り分け提案を出す。直近のユーザー発言（container 再起動 / ブラウザリロード / cache クリア等）は (b) の強い根拠として仮説確度に反映する。深掘り直行で 20〜30 分の調査ロスを避ける。既存 [#prtp-848-verify-p5]（予言・断定前のユーザー操作副作用棚卸し）の障害調査版 [#prtp-848-verify-p7]

## D-4: サブエージェント / 前セッション返却物の裏取り

サブエージェントの findings / handover / 前例パターンを受け取った時に発火。詳細は `~/.claude/knowledge/verify-rules.md` の (D-4) 節を参照。

- **handover サマリの「訂正済み・確定・決定」表現は根拠まで遡って確認する**: SessionStart 注入 handover の冒頭 3 セクションに「訂正済み・確定・決定」等の強表現があっても、書き手の確信の強さを表すだけで正しさの保証ではない。**前セッションが自分より上位モデル（Opus 等）or 同等モデルでも例外扱いしない**。訂正履歴が重なる項目（訂正1 → 訂正2 → …）は直前訂正が過剰訂正のリスク大。根拠となったコード / 実機 / ユーザー回答を Read で確認してから着手する [#prtp-825-adversarial-p1]
- **handover サマリの「未検証 / 既知挙動 / 検証方法の禁忌」行は初手で必ず注視する**: 「XXX の検証方法」「未検証」「既知挙動」を含む行は書き手の確信ではなく後続セッションの禁忌情報。コミット準備・テスト設計・実機検証手順を組み立てる前に必ず読み、後続の判断（「smoke test 通っている」等の user 発話）と照合してから着手する。前ルール ([#prtp-825-adversarial-p1] 「訂正済み・確定」) の兄弟ルール [#prtp-848-verify-p1]
- **サブエージェント / 上位モデル / 前セッションに調査・レビュー・訂正を依頼する時は、返答形式に「結論 + その結論に至った根拠 or 確度の高い仮説（file:line 引用 / 実機観察結果 / ユーザー回答等）」を必ず要求する**: 結論のみの返答は下位モデル引き継ぎ時に不要な訂正や曲解を招く。実例: PRTP-825 で Fable セッションが結論のみで根拠説明を省いた handover を残した結果、後続 Opus 4.7 セッションが訂正2 として過剰訂正を加え、Fable 再訂正で戻す往復が発生した。依頼プロンプトに「結論 + 根拠のセットで返せ」を明示するだけで下位/後続モデルの曲解余地を減らせる [#prtp-825-adversarial-p1]

## その他: 対話運用 / 成果物含意 / ツール失敗対処 / 情報管理

上記 4 クラスタに入らない独立性の高いルール。将来ここが肥大化したら別クラスタ (D-5, D-6) として独立させる。

- 一方向のレポートダンプを避け、レビュー/調査は対話の往復で進める。割り込みが入ったら再開時に「全体の指摘事項一覧」を簡潔に再提示してから個別対応に入る [#insight-20260609]
- 会話内で既に確立済みの事実を再導出しない・ユーザーが既に決定した事項を蒸し返さない・採用しない選択肢を user-facing メッセージで網羅列挙しない。検討中の選択肢は「網羅サーベイ」でなく「推奨1案 + 主要なトレードオフ」を返す（Claude Fable 5 prompting guide 由来の指針）
- Edit tool 失敗時の対処: **(1) old_string は直前の Read 結果から直接コピーする（自分で再入力しない、全角「）」を半角「)」に書き換える等の事故を防ぐ。日本語コメント多いファイル（TypeScript / PHP / Vue の対戦・タイピング系サーバコード等）は特に発生しやすいので警戒を強める）** (2) 失敗したら全角/半角・空白・引用符差・改行差を 1 分診断 → 直せそうなら 1 回だけリトライ (3) 診断つかない・特殊記号多発・**2 連続失敗**（`The required parameter 'new_string' is missing` 等＝モデル側 tool_use 生成バグの可能性、または全角・半角括弧混在ケース）のいずれかなら即別経路にピボット: (a) Write 全体書き直し（コミット対象外なら git restore で原本復元）、(b) Bash の `tee -a` / Python heredoc によるシンプル追記、(c) 並列実行中は即時中断して個別 Edit に切り替え + 各 Edit 前に直前 Read を再実行 [#cbt-622-recon-p3, #prtp-825-investigate-p8, #prtp-834-impl-p4, #prtp-848-hotfix 再発記録]
- **実装コード内のコメントに、設計ドキュメント (spec.md / feasibility.md / findings.md / tasks.md 等) のセクション識別子 (`(A-X)` `(B-X)` `(L-X)` 等) や ファイル名参照 (`spec.md §X` `feasibility.md §X` 等) を含めない**。コメントは「設計ドキュメントがない状態でも自己完結して読める」内容に保つ。理由: 設計ドキュメントは git commit せずローカル参照のみに留めるプロジェクトもあり、識別子・ファイル参照を含むと該当ドキュメントを持たない開発者にとって意味不明になる。spec.md の After サンプルコメントを転記する際は識別子・ファイル参照を除去する [#prtp-834-impl-p1]
- プロジェクト上「今はやらない」と判断した先送り事項・未解決の懸念は、各プロジェクトの memory やその場の会話に埋めず `~/.claude/backlog.md` への追記を提案する（解決したら `- [x]` に更新）。memory は「確立した知見・pitfall」用であり、未解決の宿題を置くと埋もれて追跡できなくなるため分離する。流れてよい日次の断片メモは `~/.claude/daily-notes/{当日}.md`
- **上位モデル（Fable / Opus 等）がメイン loop の時、成果物ドキュメント（spec.md / tasks.md / memory / handover / 会話中の設計判断メモ / rules 追記等）には「結論 + 根拠 + 検討して棄却した対抗案 + 棄却理由」を明示的に含める**（前ルールの書き手側セルフ版）: 上位モデルは自身の推論精度が高いため結論のみで簡潔に済ませがちだが、後続の下位モデル（Opus 4.7 / Haiku 等）が読んだ時に意図を曲解して過剰訂正を加える事故が起きる（PRTP-825 の Fable → Opus 4.7 過剰訂正 → Fable 再訂正で戻す往復が実例）。書き手側で「なぜその結論に至ったか / 何を検討して何を捨てたか」を残せば、下位モデルが読んだ時に「訂正が必要か / 意図通りか」を判断できる材料が揃う。「短くまとめて」等の圧縮指示があった場合でも、根拠と棄却理由は削らず結論説明を凝縮する方向で対応する

## 断定前の検証ルール集（引き出し）

机上推測で強い表現を使わない / grep だけで関数挙動を断定しない / サブエージェント指摘の反映前チェック（既存仕様確認・到達可能シナリオ確認等）/ 前例パターン流用前に適用条件を確認 等の検証ルール集は `~/.claude/knowledge/verify-rules.md` を必要時に Read（節構成: (D-1) 断定前 / (D-3) 依存ライブラリ / (D-4) サブエージェント指摘反映前 / (D-4) 前例流用前）。

一部の発火キーワード（「別の〇〇」「脆弱性」「致命的」「壁打ち」「松竹梅」「調査して」「原因は」等）は `~/.claude/hooks/prompt-rule-inject.sh` で自動誘導されるため、該当発話時は即座にチェック内容が注入される。それ以外のシーン（仮説評価出力時、サブエージェント返却処理時、前例流用検討時等）では明示的に verify-rules.md を Read する。

# 開発方針
- **原則(仕様駆動開発)**: 仕様ドキュメントが「唯一の正しい情報源」です。基本的には仕様を理解してからコーディングしてください。
- **タスクサイズ別パイプライン**: 発話直後に親が S/M/L を主観判定 → ユーザー確認。詳細は `rules/agent-collaboration.md`「派生パイプライン: 小規模タスク」参照。
    - **Sサイズ**（バグ修正 / リファクタ / 単純な変更 = 従来「バイブコーディング」相当）: 仕様駆動に拘らずユーザー指示優先で親直接Edit。実装後は `@code-reviewer` 必須
    - **Mサイズ**: spec軽版 → coder → code-reviewer → security-reviewer
    - **Lサイズ**: フルパイプライン（`rules/agent-collaboration.md` 並列実行パターン参照）
- 開発中機能の仕様ドキュメントは`@docs/plans`以下にあります。
- 過去に開発した機能の仕様ドキュメントは`@docs/pastplans`に保管されています。これらは仕様ドキュメント作成や実装の参考情報として利用してください。
- **仕様書・タスク書の配置ルール**:
    - 対象プロジェクトに `docs/` ディレクトリがある場合: `docs/plans/{YYYYMMDD}_{ブランチキー}/`
    - 対象プロジェクトに `docs/` ディレクトリがない場合:
      - **作業中**: プロジェクト直下 `.claude-task/{YYYYMMDD}_{ブランチキー}/` に配置する。サブエージェントが直接 Read できるためインライン全文渡しを回避でき、親のコンテキスト圧迫を抑えられる
      - **完了時（実装フェーズの `/retrospective` 直後のみ）**: `~/.claude/plans/{プロジェクト名}/{YYYYMMDD}_{ブランチキー}/` へ移動してアーカイブする。設計フェーズの `/retrospective` ではアーカイブしない（直後の実装セッションが `.claude-task/` を参照するため）。判定は retrospective skill §7 で 1 問確認
      - リポジトリ汚染が気になる場合は `.gitignore` に `.claude-task/` を手動追加する
    - 命名規則（spec.md, tasks.md, tests.md等）はどちらの場合も同一とする。

# コードレビュールール

## ブランチ全体レビュー時の基点選定
- ブランチ全体のコードレビューを行う際は、`develop`や`develop_XXX`の最新ではなく、**ブランチの派生元コミット**を基点にする
- `git merge-base <対象ブランチ> <ベースブランチ>` で派生元を特定する
- ユーザーから基点コミットの指定がある場合はそれに従う
- 理由: ベースブランチの最新と比較すると、他ブランチでマージ済みの変更がノイズとして混入し、レビュー精度が下がる

## キュージョブのレビュー観点
- `ShouldQueue`を実装するJobの`$timeout`と、`config/queue.php`の`retry_after`の整合性を確認する
- `retry_after` は `$timeout` より大きくなければならない（小さいとジョブ実行中に重複起動されるリスクがある）
