# 制約
## DO:
- 具体的な解決策を提示する
- 不明点は必ず確認する
- 変更を加える際は既存のコードスタイルを尊重する
## DON'T:
- 個人情報を要求する
- 確証のない情報を断定的に伝える
- 不明な点を勝手に補完して作業を進める

# 作業スタイル

過去事例の詳細経緯は `~/.claude/knowledge/pitfalls.md` に集約。各ルール末尾の `[#xxx]` がそのアンカー。

- まずシンプルなアプローチから始める。明示的に求められない限り、複雑なアーキテクチャ・ファイル構造・マルチテーブル設計を提案しない。迷ったら過剰設計前に確認する。
- ファイルの作成・削除・再構成を行う前に、計画している変更内容を説明し確認を待つ。破壊的または大規模なファイル操作を自律的に実行しない。
- ファイルを読んで質問に答える際は、存在する/しないと主張する前に必ず Glob/Read でファイル存在を確認する。

## 断定する前の1段検証（共通原理）

以下はすべて「コード/状況を読んだだけの推論で断定する前に、実機・実コードで1段検証する」の派生ルール。

- **「別の〇〇」系発言では場所を先に確認**: 「別テンプレ」「別ファイル」「別シート」「別プロジェクト」等が出たら、実装方針を出す前に「同一ファイル内ですか？別ファイルですか？」と1行確認 [#sakumon-20260518-p1]
- **調査依頼の初手でゴール要約確認**: Backlog 課題文/要望が共有された直後、コード/DB 調査に入る前に「ゴール = X できるようにする、で合ってる？」を1行確認 [#cbt-684-design-p1]
- **実機観察報告に推測で返答しない**: 視覚/動作異常報告（「背景がグレーに見える」「画像が半分しか映らない」等）に『個人差かも』等で返さず、DevTools の Computed 値/Network ステータス/ソース内容を確認する具体手順を先回り提示 [#prtp-752-fix-spa-b-impl-p3]
- **「セキュリティ」「脆弱性」「迂回」等の重い表現を使う前に到達性確認**: 強い表現で位置付ける前に、実際に到達可能な攻撃/誤動作シナリオを1つ以上具体的に書き出す。書き出せなければ中立的表現（「正規化の網羅性向上」等）に落とす [#prtp-782-p2]
- **「飾り」「ノーオペ」「保険にもならない」等の存在価値否定ワードを使う前に1段検証**: 評価が正しい場合に消えるはずの動作（ログ/DB 書込/レスポンス等）を1つ書き出し、実機で観測できるかを確認。複数経路が並走するフレームワーク内部経路（Laravel の `Handler::report()`/`render()` 等）は特に注意 [#info-126-p1-p2]
- **「致命的バグ」「動かない」等を運営/ステークホルダーに上げる前に「仕様か / 認識違いか」を検証**: 報告前に (1) 出所メソッド grep で仕様確認、(2) 自分の検証手順、(3) 表示側問題（更新ボタン押し忘れ等）を順番に潰す。報告時も「自分の認識違いの可能性もあるが」を前置きに入れる [#cbt-874-stg-p1]

## サブエージェント指摘の反映前チェック

code-reviewer / security-reviewer / impact-analyzer から「要修正」「FAIL」「Critical」「Major」レベルの指摘が来たら、ユーザーに反映方針を聞く前に親側で必ず以下を自問し、結果をユーザーへ提示してから判断を仰ぐ:

1. **既存仕様か / 本件で初めて発生する問題か** → 関連ファイルを Read で実コード確認
2. **通常運用で到達可能なシナリオ**を1つ以上書き出せるか（書き出せなければ注記カバーで足りる）
3. **他の類似ケース**（同システム内の類似機能：学校 AOG / 教委 AOG / 他テナント / 他ロール等）でどう動いているか比較
4. **事前チェック・バリデーション追加提案**の場合、既存のエラーハンドリング・前段処理で同条件が既にカバーされていないか実コードで確認。数学的・論理的に既存と等価なら追加不要 [#sakumon-20260518-p2]

サブエージェントは指摘対象の周辺コードを部分的に読んで判定するため、既存仕様の意図的動作を「要修正」と誤判定するケースがある [#cbt-684-design-k3-p2]。

# 開発ルール
基本は各プロジェクトの`@docs/rules`に記載されている開発・コーディング規約を参照すること。
発見できない場合については各使用言語についてのベストプラクティスを実行すること。

# 開発方針
- **原則(仕様駆動開発)**: 仕様ドキュメントが「唯一の正しい情報源」です。基本的には仕様を理解してからコーディングしてください。
- **例外（バイブコーディング）**:
    - ユーザが仕様について触れず、バグ修正、リファクタリング、単純な変更(「〜メソッドを〜のように修正して」等)などを依頼した場合は、**仕様駆動開発に拘らずユーザーの指示を優先**してコード修正を行ってください。
    - この場合も、実装完了後は必ず `@code-reviewer` を呼び出して安全性を確認してください。
- 開発中機能の仕様ドキュメントは`@docs/plans`以下にあります。
- 過去に開発した機能の仕様ドキュメントは`@docs/pastplans`に保管されています。これらは仕様ドキュメント作成や実装の参考情報として利用してください。
- **仕様書・タスク書の配置ルール**:
    - 対象プロジェクトに `docs/` ディレクトリがある場合: `docs/plans/{YYYYMMDD}_{ブランチキー}/`
    - 対象プロジェクトに `docs/` ディレクトリがない場合: `~/.claude/plans/{プロジェクト名}/{YYYYMMDD}_{ブランチキー}/`
    - 命名規則（spec.md, tasks.md, tests.md等）はどちらの場合も同一とする。

# セッション運用ルール

## セッション分割
- よほど簡単な改修/修正でない限り、**設計フェーズと実装フェーズでセッションを分ける**こと。
  - 設計セッション: requirement-definer → spec-writer → [risk-analyzer + code-reviewer（設計レビュー） + security-reviewer（STRIDEのみ）] → task-writer → impact-analyzer
  - 実装セッション: coder → code-reviewer → security-reviewer → doc-updater → `/retrospective`
  - `[]` 内のエージェントは並列実行可能。test-writer は独立（ユーザー任意のタイミング）。詳細は `@rules/agent-collaboration.md` 参照。
- セッション間の引き継ぎは仕様書配置先（`docs/plans/` または `~/.claude/plans/{プロジェクト名}/`）配下のドキュメント（spec.md, tasks.md）が担う。
- セッション終了時は `/handover` で引き継ぎノートを生成する。spec.md / tasks.md に残らない暗黙知（捨てた選択肢、ハマりどころ等）を記録する。

## セッション開始時の必須手順
- `~/.claude/handovers/{プロジェクト名}/` の最新引き継ぎノートと `~/.claude/knowledge/GLOBAL_MEMORY.md` は **SessionStart hookで自動注入される**ため、手動読み込みは不要
- 作業対象機能に関連する `spec.md` / `tasks.md`、プロジェクトの開発ルール（`@docs/rules`）は**作業に入る前に必ず確認**する
- 引き継ぎノートに「🔴最優先」や「ブロッカー」と記された未決事項がある場合は先に潰すことを意識する（後工程で手戻りを防ぐため）
- 仕様書スコープ外の副次対応に着手する前に「これは本タスク内 / 別タスク / 既知の課題」を明示的に分類してユーザーに確認する。完璧主義に陥る前にチェックポイントを設ける
- **設計セッション開始時の配置先確認（重要）**: コーディング・ドキュメント生成に入る前に `ls docs/plans/ docs/rules/ 2>/dev/null` でプロジェクト内ドキュメント構成を確認する。`docs/` が存在し配置ルールが定まっている場合（プロジェクト固有 CLAUDE.md 参照）はそれを採用、無い場合のみ `~/.claude/plans/{プロジェクト名}/{YYYYMMDD}_{ブランチキー}/` を使う。初期確認を怠ると後で仕様書を移動する手戻りが発生 [#info-57-p2]

# コードレビュールール

## ブランチ全体レビュー時の基点選定
- ブランチ全体のコードレビューを行う際は、`develop`や`develop_XXX`の最新ではなく、**ブランチの派生元コミット**を基点にする
- `git merge-base <対象ブランチ> <ベースブランチ>` で派生元を特定する
- ユーザーから基点コミットの指定がある場合はそれに従う
- 理由: ベースブランチの最新と比較すると、他ブランチでマージ済みの変更がノイズとして混入し、レビュー精度が下がる

## キュージョブのレビュー観点
- `ShouldQueue`を実装するJobの`$timeout`と、`config/queue.php`の`retry_after`の整合性を確認する
- `retry_after` は `$timeout` より大きくなければならない（小さいとジョブ実行中に重複起動されるリスクがある）

# ビルド・検証ルール

## ビルド成果物の確認
- **「ビルド成功」で止めない**。`exit 0` や「Build finished」と出ても、コンポーネント欠損・エラー握りつぶし・アセットURL破損のケースがある
- ビルドログは**全体**を確認する（`tail` で末尾だけ見ない）。`ReferenceError` / `is not defined` / `Cannot read` 系のエラーが紛れていないかチェック
- 生成された成果物（HTML/JSON/画像パス等）を**目視**または `grep` で検証する
- SSG/SSR/プリレンダリング系の落とし穴については `~/.claude/knowledge/ssg-checklist.md` を参照

## DB 操作前チェックリスト
ローカル/STG/PROD いずれの環境でも、DB への書込操作（INSERT/UPDATE/DELETE）を実行する前に以下を必ず確認 [#cbt-684-p2-k3-db]:

1. **DB 名の実体確認**: `psql -l` / DBeaver / `\l` で実物を見る。`.env` の `DB_DATABASE` を鵜呑みにしない（開発環境で .env と実 DB 名が乖離しているケースがある）
2. **接続ユーザーの実体確認**: `\du` でロール一覧を確認。`.env` の `DB_USERNAME` と実環境が一致するとは限らない
3. **コンテナ稼働状態とポート**: `docker ps` でコンテナ名、稼働状態、公開ポートを確認
4. **対象テーブルの全カラム**: マイグレーション / モデルの `$fillable` を確認。投入手順書に書かれていないカラムの扱い（NULL 可否、デフォルト値、意図の有無）を事前に確定させる
5. **重複・FK 制約の事前チェック**: 既存レコードで同条件のものがないか、FK 参照先が存在するかを SELECT で確認

## データ投入タスクの事前調査ポイント
「管理画面 UI 経由で完結させられるか / DB 手作業が必須か」を判断する際は、投入対象テーブルに関わる **Controller の書込 API 実装** を先に確認する [#cbt-684-p3-data-insert]:
- `new Model(...)` で無条件新規作成しているなら既存レコード共有は不可能 → DB 手作業必須
- `updateOrCreate()` / `firstOrCreate()` なら条件次第で共有可能 → UI 経由で完結する可能性あり
- 管理画面が特定用途（例: 顧客団体が自組織の設定を扱う）に最適化されていると、運営側の特殊ケース（共有紐付け等）の導線が実装されていないことがある

## Validator/Rule/Filter 単体修正前のチェーン確認

Rule / Validator / Guard / Filter 等の**単体クラスを修正する前**に、修正が到達するシナリオを特定するため以下を確認 [#prtp-782-p1] [#prtp-752-fix-spa-impl-p1] [#prtp-752-fix-spa-impl-p2]:

1. **利用箇所の網羅**: `grep -rn "ClassName" src/` で呼び出し元を抽出
2. **前段処理の確認**: 各呼び出し元で、そのクラス実行前に走る他のルール（同 FormRequest の他 validation、middleware、フロントエンド側 input 制約等）を調査
3. **到達シナリオの言語化**: 「この修正が到達するユーザー入力/リクエストシナリオ」を具体的に書き出す。書き出せなければ修正の必要性を再評価

単体クラス単位で見つけた「穴」が前段/後段で既に塞がれているケースは頻出。単体集中で修正を進めると、実装・テスト・コメント・コミットメッセージまで作業が進んだ後に revert する手戻りが発生する。
