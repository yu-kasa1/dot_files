# 学習ログ (study-log)

## 概要
知らないリポジトリ・ライブラリ・実装パターンを探索した後に、得られた知見を構造化してNotionに投入する。
単なる学習ログではなく、**自分の今後の設計・実装に活かせる引き出し**として蓄積することを目的とする。

## retrospectiveとの違い
- **retrospective**: Claudeとの協業プロセス改善（KPT）。ローカル記録。フロー・プロンプト・エージェント連携の材料。
- **study-log**: 外部リポ探索で得た技術的知見。Notion投入。設計の引き出し。

両者は観点が違うので、学習探索セッションでは**両方実行してよい**。
同じ出来事でも切り口が変わる（例: 「grep連発で時間を食った」はretrospective、「このリポのDIパターンが参考になる」はstudy-log）。

## 呼び出しタイミング
- 外部リポジトリ・OSSの探索や調査を行った後
- 知らない技術スタック・パターンを学習した後
- ユーザーが明示的に呼び出した時

## 実行手順

### 1. 投入先Notionページの確認
**都度ユーザーに投入先を確認する**。プロジェクトごとに固定の投入先は持たない。

- ユーザーに「投入先のNotionページ（親ページID or URL）」を質問する
- ユーザーから指定された親ページ配下に新規ページを作成する

### 2. 探索内容の整理
セッション内の会話コンテキストから、以下を整理する:

- **対象リポジトリ**: 名前とURL（分かれば）
- **探索範囲**: 読んだファイル・ディレクトリ・機能
- **学んだこと**: 実装パターン、設計判断、使われているライブラリ等
- **設計に効きそうな点**: 自分の今後の作業への示唆（重要）
- **未解決の疑問**: あとで調べたい点
- **タグ**: 技術スタック・テーマ等

情報が足りない場合はユーザーに確認する。勝手に補完しない。

### 3. Notionページ作成
`mcp__claude_ai_Notion__notion-create-pages` を使用して、指定された親ページ配下にページを作成する。

ページタイトルは `{YYYYMMDD} {対象リポジトリ名}` の形式を基本とする。

### 4. ローカルインデックスの記録
セッション開始時の参照材料として、ローカルにも軽量メタ情報を記録する。

- **記録先**: `~/.claude/study-logs/{YYYYMMDD}_{リポ名}.md`
- **配置**: フラット配置（プロジェクト別ディレクトリは作らない）
- **内容**: メタ情報のみ（本体はNotion）

ローカルファイルのテンプレート:
```markdown
# {対象リポジトリ名}

- **探索日**: YYYY/MM/DD
- **URL**: {リポジトリURL}
- **タグ**: {技術スタック, テーマ}
- **Notion**: {作成したNotionページのURL}

## 一行サマリ
{このリポから何を学んだかを1-2行で}
```

### 5. ユーザーへの報告
作成したNotionページのURLとローカルファイルのパスをユーザーに報告し、内容の過不足を確認する。

## 出力テンプレート（Notionページ本文）

```markdown
# {対象リポジトリ名}

## 基本情報
- **リポジトリ**: {名前}
- **URL**: {URL}
- **探索日**: YYYY/MM/DD
- **タグ**: {技術スタック, テーマ}

## 探索範囲
{読んだファイル・ディレクトリ・機能の範囲}

## 学んだこと
- {実装パターン・設計判断・使われているライブラリ等を箇条書き}

## 設計に効きそうな点
- {自分の今後の作業にどう活かせるか、具体的な示唆を箇条書き}

## 未解決の疑問
- {あとで調べたい点を箇条書き}
```

## 注意事項
- **投入先は都度確認する**。固定の投入先を持たない
- **要約しすぎない**。学習ログは後から選別するバッファなので、情報は少し多めでよい
- **自分の設計に効きそうな点**の欄は空欄にしない。ここが無いと単なる学習ログになり、設計の引き出しとして機能しない
- **勝手な補完をしない**。会話コンテキストに無い情報はユーザーに確認する
- Notionに投入した後の取捨選択はユーザーが行う。Claudeはあくまでバッファへの投入まで担当する
- ローカルインデックスは**メタ情報のみ**。本文を丸ごと残さない（肥大化防止）。詳細はNotionを参照する
