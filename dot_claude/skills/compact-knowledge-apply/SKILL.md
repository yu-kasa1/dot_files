# ルール群コンパクション反映 (compact-knowledge-apply)

## 概要
compact-knowledge が出力したコンパクション候補レポートを読み込み、修正案を提示・承認・反映する。`compact-knowledge` とペアで運用する単独起動専用スキル。

**反映は必ずユーザー承認を挟む**。自動素通り反映は行わない。

## 呼び出しタイミング
- compact-knowledge レポート確認後、整理を実施したいタイミング
- 「コンパクション反映」「ルール圧縮の反映」「pitfalls 整理を実施」等の発話で起動
- **weekly-bundle からは呼ばない**（バンドルは検出のみで完結させる）

## 引数
- 引数なし: `~/.claude/weekly-reports/run-log/` から **最新の** `*-compaction.md` を自動選択
  - 「最新」の定義: **ファイル名日付 (YYYY-MM-DD) の最大値**を採用（mtime ではなくファイル名基準）
  - 同日複数ファイルが存在する場合は、ファイル名の辞書順最大（例: `2026-05-21-compaction-adhoc.md` > `2026-05-21-compaction.md`）
- 引数あり（例: `2026-05-21`）: `~/.claude/weekly-reports/run-log/YYYY-MM-DD-compaction.md` を読み込む

## 反映パターン

### A. 未参照 pitfalls アンカー → 貼り忘れ修正
ルール本文に **散文出典**（例: 「cbt-p1-server CBT-541 設計 retrospective P-2」）が記載されているが、アンカー形式 `[#cbt-541-design-p2]` で貼られていないケース。

**反映手順**:
1. pitfalls.md の該当アンカーの内容を読む
2. CLAUDE.md / agents/*.md / rules/*.md を grep で探索し、散文出典に対応する記述を特定する
3. 該当行末尾に `[#anchor-name]` を追記する修正案をユーザーに提示
4. 承認後 Edit で反映

### B. 未参照 pitfalls アンカー → 削除
ルール本文にも痕跡なし、かつ陳腐化していると判断したケース。

**反映手順**:
1. pitfalls.md の該当エントリ全文（見出し + 本文）をユーザーに提示
2. ユーザーに「削除して良いか」を確認
3. 承認後、pitfalls.md から該当エントリ + 索引行を削除

### C. GLOBAL_MEMORY / memory/*.md 重複候補 → 統合
複数行が同一観点を扱っており1行に統合可能なケース。GLOBAL_MEMORY.md 鉄則セクション内、memory/*.md 同一 topic 内、または memory/*.md ファイル横断のいずれも対象。

**反映手順**:
1. 統合対象の行2件（以上）を Read で確認
2. **統合後の文面案を Claude が作成**し、ユーザーに提示
3. 承認後、元の行を削除し、統合後の行を追加（妥当な配置先を選択：鉄則 / 元のtopic / 別topicへの移動）
4. 出典は両方残す形で記載（例: `（INFO-34, INFO-126）`）
5. **ファイル横断統合の場合**: 配置先の妥当性をユーザーに明示確認（例: design.md と process.md にまたがる重複を design.md に統合する判断）

### D. 複数参照アンカー → 対象外
意図的な複数参照（同じ pitfall が複数エージェントに関連）のケースが多いため、本スキルでは反映対象外。レポートで列挙のみされている場合は何もしない。

ただしユーザーが「これは統合したい」と明示した場合のみ、pitfalls.md のエントリを統合する形で反映する。

## 実行手順

### 1. 事前チェック
- `git status` で未コミット変更を確認。あれば「未コミット変更があります。コミットしてから実行することを推奨します。続行しますか?」と確認
- 対象レポートファイルの存在確認

### 2. レポート読み込みと候補分類
レポートの A/B/C 各セクションを読み、反映パターン（A.貼り忘れ修正 / B.削除 / C.統合 / D.対象外）に分類する。

### 3. 修正案の提示
候補ごとに以下の形式で修正案を提示する。パターン別にフォーマットが異なる:

**パターン A（アンカー貼り忘れ修正）**:
```
【候補 1/N】 (種別: アンカー貼り忘れ修正)
- 対象アンカー: cbt-541-design-p2
- 修正対象: rules/agent-collaboration.md:144
- 現状: 「...cbt-p1-server CBT-541 設計 retrospective P-2 より」
- 修正後: 「...cbt-p1-server CBT-541 設計 retrospective P-2 より [#cbt-541-design-p2]」
```

**パターン B（エントリ削除）**:
```
【候補 2/N】 (種別: エントリ削除)
- 対象アンカー: xxx-yyy-zzz
- 削除対象1: pitfalls.md L42-L48（## xxx-yyy-zzz セクション全体）
- 削除対象2: pitfalls.md L8（索引の `- [#xxx-yyy-zzz](#xxx-yyy-zzz) — ...` 行）
- 削除前の本文（参考表示）:
  > {pitfalls.md の該当エントリ全文}
- 削除根拠: ルール本文に痕跡なし、かつ陳腐化判定
```

**パターン C（ルール統合）**:
```
【候補 3/N】 (種別: ルール統合)
- 統合対象: GLOBAL_MEMORY.md L95 + L119
- 現状1: 「曖昧な短質問（「直せる？」等）は…意図確認（INFO-34 design）」
- 現状2: 「「調査だけ」「確認だけ」明示タスクで修正示唆を受けたら…メタ確認（INFO-126）」
- 統合後（提案）:
  > 「ユーザー指示が曖昧（「直せる？」等）／スコープ拡張（「調査だけ」→修正示唆等）の場合、行動前に意図確認1行を挟む（INFO-34, INFO-126）」
- 配置先: §プロセス（既存位置を引き継ぐ、両元行は削除）
```

### 4. ユーザー承認
候補ごと個別に承認を取る、または「全部承認」「個別に確認」を選ばせる。AskUserQuestion で:

- 承認方式: 一括承認 / 個別に承認 / キャンセル

ルール統合（パターン C）は必ず個別承認を強制する（文面書き直しを伴うため）。

### 5. 反映実施
承認された修正のみ Edit で反映する。複数ファイルにまたがる場合は順次実施。

### 6. 変更サマリ出力
反映後に以下を出力:

```
## 反映サマリ

### 反映済み（N件）
- [rules/agent-collaboration.md:144](dot_claude/rules/agent-collaboration.md#L144) — アンカー貼り忘れ修正（#cbt-541-design-p2）
- [knowledge/GLOBAL_MEMORY.md:95-119](dot_claude/knowledge/GLOBAL_MEMORY.md#L95) — ルール統合（意図確認系）

### スキップ（N件）
- ...（理由付き）

### 次のアクション
- `git diff` で変更内容を確認
- 問題なければ commit
```

## 注意事項
- **必ずユーザー承認を挟む**: 自動素通り反映は禁止。特にパターン C（統合）は必ず個別承認
- **未コミット変更を尊重する**: 既存の未コミット変更がある状態では警告を出す。混ざると revert が困難になる
- **シンボリックリンクに注意**: `~/.claude/` と `dot_files/dot_claude/` はシンボリックリンク済み。どちらのパスで Edit しても同じファイルを指す。ただし `git status` / `git diff` は **`dot_files/` 側で実行**する（`~/.claude/` 直下は git リポジトリではないため）
- **削除は慎重に**: パターン B（削除）は復元が pitfalls.md からの再記述になるため、必ずユーザーに「本当に削除して良いか」を明示確認する
- **反映後の検証はユーザー側**: 本スキルは `git diff` 確認を促すのみで、自動 commit はしない
- **weekly-bundle からは呼ばない**: バンドル内で反映まで自動化すると素通り運用と矛盾する。本スキルは単独起動専用
