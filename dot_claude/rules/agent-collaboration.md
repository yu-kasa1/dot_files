# サブエージェント連携

## 並列実行パターン

依存関係がないエージェントは並列実行してよい。以下が主要な並列実行ポイント。

### 設計フェーズ前段（オプショナル）: feasibility-writer

```
ユーザー要求あり
   │
   ▼
親エージェント: 「道筋が立っているか」4項目チェック
   ├── #1 親が初手で具体的なステップ（1-2-3）を書けない
   ├── #2 外部リソース（AWS/API/ライブラリ等）の制約確認が必要
   ├── #3 複数のアーキテクチャ選択肢が初手で浮かぶ
   └── #4 ユーザーが「実現可能性」「壁打ち」「松竹梅」「どっちがいいか」等を発話
   │
   ├── いずれか該当 → ユーザー確認後 feasibility-writer 起動
   │       → feasibility.md 初期ドラフト生成
   │       → ユーザー回答を親エージェントが Edit で追記
   │       → 方針確定後、requirement-definer (省略可) → spec-writer へ
   │
   └── いずれも該当せず → requirement-definer 直行
```

- feasibility-writer は **オプショナル**。要件が明確で選択肢比較不要なら呼ばない
- 「選択肢が松竹梅でメリデメが分かれる吟味」が必要な時に使う
- 「現状整理」「リスク分析」「STRIDE」は別ドキュメント / 別エージェントの責務、ここでは選択肢比較に専念
- 200〜300 行を目安、肥大化させない

### 設計フェーズ: spec-writer完了後
```
spec-writer 完了
  ├── risk-analyzer（FMEA分析）  ← spec.md を読んで独立にリスク分析
  ├── code-reviewer（設計レビュー） ← spec.md を読んで独立にレビュー
  └── security-reviewer（STRIDE脅威分析） ← spec.md を読んでセキュリティ観点の脅威分析
全エージェント完了後 → task-writer
```
- 3エージェントとも spec.md を入力とし、互いに依存しない
- security-reviewer は設計段階では **STRIDEチェック（ST-01〜06）のみ** 実施。OWASPチェックはコード実装後に実施する
- 全エージェントの結果をユーザーが確認した後、必要な修正を spec.md に反映してから task-writer へ進む

### 実装フェーズ: task-writer完了後
```
task-writer 完了
  └── impact-analyzer（影響範囲調査）  ← tasks.md + 既存コードベースを読んで波及範囲を調査
影響分析の結果をユーザーが確認後 → coder
```
- task-writer が作成したタスク一覧に対して、実装着手前に影響範囲を洗い出す
- 影響分析の結果により、タスクの追加・修正が必要な場合は tasks.md を更新してから coder へ進む

### 実装フェーズ: coder全タスク完了後
```
coder 全タスク完了
  └── code-reviewer（コードレビュー）  ← コード + spec.md を読んで品質チェック
code-reviewer 完了・修正反映後
  └── security-reviewer（セキュリティレビュー） ← コードを読んでセキュリティ特化チェック
全完了後 → 修正対応 → doc-updater
```
- code-reviewer → security-reviewer は直列実行。品質レビューの修正を反映した状態でセキュリティチェックを行い、観点のブレを防ぐ
- test-writer はこのフローとは独立。ユーザーが任意のタイミングで呼び出す
- レビューで修正が発生した場合、修正後に該当チェック項目のみ再判定すればよい（全部やり直し不要）

### 並列実行時の注意事項
- 並列実行するエージェントの結果はそれぞれ独立してユーザーに提示する
- 修正の適用順序はユーザーが判断する
- 並列実行はあくまで効率化手段。ユーザーが順次実行を希望する場合はそれに従う

## ドキュメント生成系サブエージェントの成果物受け取り

spec-writer / task-writer / impact-analyzer など**ドキュメント生成を行うサブエージェント**および coder / doc-updater / test-writer などコード生成を行うサブエージェントは、起動プロンプトに **「成果物は親側で書き込むため、全文（または差分）をメッセージ本文で返してください。Write/Edit ツールは利用不可」** を明記すること。

### 理由
全サブエージェント定義（`~/.claude/agents/*.md`）の frontmatter に `tools: Read, Glob, Grep, Bash` を明示しており、Write/Edit は意図的に外している。書き込みは必ず親エージェントを経由させ、ユーザーがレビューできるポイントを担保するため。読み取り系（Read/Glob/Grep/Bash）は許可済みなのでサブエージェント実行中の読み取り承認プロンプトは発生しない。

### Bash 経由の書き込みについて
Bash は許可されているが、`echo > file` / `sed -i` / `rm` / `mv` / `cp -f` 等の書き込み・削除系コマンドは **(1)** 各エージェント定義の「## ツール利用制約」セクションで明示禁止、**(2)** `~/.claude/settings.json` の `permissions.deny` で deny 指定（`rm`, `sed -i`, `mv`, `cp -f`, `git reset --hard` 等）、の二段で抑止している。リダイレクト（`>` / `>>` / `tee`）等パターンマッチで完全に止められないものは (1) のソフト制約に依存。

### 返送内容の検査（重要）
ドキュメント生成系サブエージェントの返送には **HTML エンティティエンコード**（`&lt;` / `&gt;` / `&amp;`）が混入することがある。Write 前に以下を確認:
- 返送本文に `&lt;` / `&gt;` / `&amp;` が含まれないか目視 or `grep` でチェック
- 含まれていたらデコードしてから Write（特に `<script>` / `<div>` / `<head>` 等の HTML タグを多用する仕様書で頻発）
- 気づかず Write すると壊れたドキュメントが commit され、後続の coder が誤読する（PRTP-752-fix-spa retrospective より）

### 大規模成果物（100 行超）受け取り時のフロー（重要）

サブエージェントが **100 行を超える成果物**（spec.md / tasks.md / 大規模リスク分析レポート等）を返してきた場合、AskUserQuestion で全文確認を求めない。チャット UI のレンダリング上、長文返答はユーザーが読めない／省略される／スクロール負荷が高い。代わりに以下のフローを定型化する:

1. **即ファイル書き込み**（プロジェクト外なら `~/.claude/plans/{プロジェクト}/{YYYYMMDD}_{ブランチキー}/` 配下、配置ルールは CLAUDE.md「仕様書・タスク書の配置ルール」に従う）
2. ユーザーへ「**ファイルパス（クリック可能なリンク） + 構造サマリ（章立て + 各章の要点 3-5 行）**」のみ提示
3. 「内容確認後、書き込み内容で次フェーズへ進めて良いか」を AskUserQuestion で確認

「全文をそのまま貼って `この内容で書き込みますか？` と聞く」のは失敗パターン（cbt-p1-server CBT-684 設計 retrospective P-1 より。500 行の spec.md 全文を AskUserQuestion で確認させ「どの内容ですか見えません」と返ってきた）。100 行未満の成果物は従来通り全文提示で構わない。

## サブエージェントへの仕様書受け渡し

サブエージェントはプロジェクトルート配下のファイルのみ読み取り可能。`~/.claude/plans/` 等プロジェクト外に仕様書を配置しているケースでは、サブエージェントが直接ファイルを読めない。

### ルール
- **プロジェクト内に仕様書がある場合**（`docs/plans/` 等）: ファイルパスを指示すればサブエージェントが自分で読める。追加対応不要
- **プロジェクト外に仕様書がある場合**（`~/.claude/plans/{プロジェクト名}/` 等）: 親エージェント（呼び出し元）が仕様書の内容を読み取り、**サブエージェント起動時のプロンプトにインラインで全文を含めて渡す**
- **要約・抜粋は原則禁止**。情報欠落による手戻りリスクが高いため、全文をそのまま渡す
- **大規模仕様（800行超）で分割済みの場合**: そのエージェントのタスクに関連するファイルのみ渡す。ただし関連ファイルは全文で渡すこと

### 渡し方のフォーマット
```
以下は仕様書（spec.md）の全文です:

---spec.md---
（spec.mdの全文）
---end---

上記仕様に基づき、〜を実施してください。
```

## db-analyzer
既存のデータベーススキーマを分析し、テーブル構造・リレーション情報を提供するエージェントです。
### 呼び出しルール
- **仕様策定時**: `@spec-writer`がDB設計を行う際に、既存スキーマ調査が必要な場合に呼び出します。
- **実装時**: `@coder`がマイグレーション/モデル実装を行う際に、既存テーブル構造の確認が必要な場合に呼び出します。
- **単独呼び出し**: DB構造を把握したい場合に直接呼び出すことも可能です。
- **対応フロー**:
  1. 調査目的・対象範囲を確認
  2. マイグレーション/モデルファイルを調査
  3. スキーマ分析レポートを作成
  4. ユーザー確認後、必要に応じて`@spec-writer`または`@coder`へ引き継ぎ
- **重要な制約**: Dockerコンテナ上のDBのみ参照可能（本番・ステージング環境は絶対禁止）

## risk-analyzer
仕様書に対してFMEA（故障モード影響解析）を適用し、設計段階でリスクを洗い出すエージェントです。
### 呼び出しルール
- **仕様作成完了後**：`@spec-writer`の完了後、`@code-reviewer`（設計レビュー）と並列で呼び出します。
- **単独呼び出し**：既に仕様が存在する場合は、直接`@risk-analyzer`を呼び出すことも可能です。
- **対応フロー**：
  1. 仕様書（spec.md）を読み込み
  2. 各処理フローに対してFMEA分析を実施（深刻度 x 発生頻度 x 検出難易度 = RPN）
  3. リスク分析レポートを作成
  4. ユーザー確認後、spec.mdへのフィードバックおよびtasks.mdへのタスク追加を提案

## impact-analyzer
変更予定の内容に対して、既存コードベースへの波及範囲を調査するエージェントです。
### 呼び出しルール
- **タスク作成完了後**：`@task-writer`の完了後、`@coder`着手前に呼び出します。タスク一覧に対して影響範囲を事前に洗い出す。
- **リファクタリング時**：影響範囲が広い変更を行う前に呼び出します。
- **単独呼び出し**：既存機能の修正・バグ修正で、影響範囲を把握したい場合に直接呼び出すことも可能です。
- **対応フロー**：
  1. 変更内容を把握（何を、どう変えるか）
  2. コードベースから直接的・間接的な影響範囲を調査
  3. 影響レベル（要修正 / 要確認 / 影響なし）を分類してレポート
  4. 必要に応じて追加タスクを提案し、tasks.md を更新
- **起動プロンプトで明示すべき観点（現存データ実機確認、重要）**：
  - 「**改修対象テーブルの既存レコードを実 DB で SELECT し、設計前提と矛盾していないか検証すること**。特に admin / system / マスタ系で固定的なデータを扱うテーブル、認証 / SAML / 動的テーブル系では、過去のプロト用途で作成された既存レコード（global_id 非同値の AOG、孤児になった動的テーブル等）が残置している可能性が高い。検証結果を impact-analyzer レポートに『§ 現存データ実機確認』セクションとして必ず含める。プロジェクト外（STG/PROD）の確認は依頼ベースで OK だがローカルは impact-analyzer が直接 Bash で SELECT 実行する」（cbt-p1-server CBT-684 実装 retrospective P-1 より。設計時に admin AOG が存在しない前提で書かれていたが、実際にはプロト AOG が 2026-04-21 作成済みで残置 → 実装段階で発覚しクリーンアップ手順を急遽追加、spec.md / tasks.md 約 100 行追記の手戻り発生）

## feasibility-writer
実装方式・アーキテクチャに複数案がある設計フェーズで、選択肢比較と論点整理を行うエージェントです。requirement-definer / spec-writer より前段の「壁打ちフェーズ」で使用します。

### 呼び出しルール
- **壁打ちフェーズ**: requirement-definer / spec-writer の前段。設計の方針自体が未確定で複数案の比較が必要な場合に使用
- **発動条件（OR、いずれか1つでも該当）**:
  - 親エージェントが「道筋が立っているか」4項目チェックで該当判定 → ユーザーに確認後起動:
    - 親が初手で具体的なステップ（1-2-3 形式）を書けない
    - 外部リソース（AWS / 外部 API / ライブラリ / インフラ）の制約・提供形式の確認が必要
    - 複数のアーキテクチャ選択肢が初手で浮かぶ
    - ユーザー自身が「実現可能性」「方向性」「壁打ち」「松竹梅」「どっちがいいか」等を発話
  - ユーザーが明示的に「壁打ちしたい」「実現可能性から相談したい」等を発話 → 親判定スキップして起動可
- **呼び出さない条件**: 既存パターン踏襲 / 1ファイル局所修正 / 単純な仕様確認のみ / 方針確定済み / 軽微なバグ修正
- **対応フロー**:
  1. ユーザー要求と既存コードベース、必要なら外部リソース（GitHub Releases、AWS API 等）の実機確認
  2. 選択肢の網羅的洗い出し（案 A / B / C / D 等）
  3. 比較表（メリ / デメ / コスト / 既存システム整合性）と推奨案を提示
  4. ユーザー判断質問（最優先 3〜5 問）を整理
  5. 前提確認事項（事前調査タスク化候補）を列挙
  6. ユーザー回答による §決定事項 の追記は親エージェントが Edit で対応（feasibility-writer 再呼び出し不要）
  7. 方針確定後、必要なら `@requirement-definer`（要件整理）→ `@spec-writer` へ
- **配置**: `docs/plans/{YYYYMMDD}_{ブランチキー}/feasibility.md`（または `~/.claude/plans/{プロジェクト名}/...` 配下、CLAUDE.md 配置ルールに従う）
- **サイズ**: 200〜300 行を目安、肥大化させない
- **起動プロンプトで明示すべき観点**:
  - 「**現状整理の詳細・リスク分析・STRIDE 脅威分析は別ドキュメント / 別エージェントの責務**、ここでは選択肢比較に専念」
  - 「**推測で選択肢を作らない**: 外部リソース（ハッシュ提供形式、API 仕様、ライブラリ機能等）は実機確認（`curl` / `gh api` 等）してから比較表を組み立てる」（keycloak-test INFO-57 retrospective より。GitHub Releases の SHA256 提供を前提に仕様化しかけたが、実機確認で SHA1/MD5/GPG のみ提供と判明）
  - 「**手動配置・初期セットアップ依存の案を出す場合は正当化を明示**: 自動化経路（CodeDeploy 環境変数 / アーティファクト同梱 / IMDS / Parameter Store 等）が無い理由を書く。安易に『ユーザーが事前配置する前提』で済ませない」

## requirement-definer
ユーザーの要望を整理し、仕様作成に必要な要件を定義するエージェントです。
### 呼び出しルール
- **新機能開発時**：ユーザーから新機能の開発依頼があった場合、`feasibility-writer` の発動条件に該当しないことを確認のうえ、`@requirement-definer`を呼び出して要件を整理してください。該当する場合は先に `@feasibility-writer` を起動して方針確定してから要件定義に進みます。
- **対応フロー**：
  1. ユーザーの要望をヒアリング
  2. 要件を整理した後、ユーザー承認後に自動的に`@spec-writer`を呼び出し
- **呼び出し前の親側チェック（重要）**：要件定義の「ユーザー確認事項」に、既存ファイルの形式・値に依存する質問（「既存フォーマットに合わせるか」「既存エントリの lastmod はどうするか」等）を含める場合、**親エージェント側でその既存ファイルを Read してから質問を組み立てる**こと。前提未確認のまま質問を作ると、ユーザー回答後に前提誤りが発覚して往復コストが発生する（PRTP-752-fix-spa retrospective より）
- **色・ビジュアル系の要件でのコントラストチェック（重要）**：色を含む UI 要件では、背景色/背景画像とスピナー/テキスト等の前景要素のコントラストが十分か確認すること。**同系色の組み合わせ（オレンジ背景+オレンジスピナー等）は視認性が著しく低下する**。要件定義段階でユーザーに「この背景にこの色で見えますか？」を確認する（PRTP-752-fix-spa B チケット設計 retrospective より。ランキング背景にオレンジスピナーで視認不可→白に変更）
- **インフラ設定変更を含むチケットでの「関連設定の全量確認」（重要）**：CloudFront / ALB / Route53 / SG / S3 バケットポリシー等**インフラ設定の変更を含むチケット**では、変更対象の設定だけでなく **同リソース内で関連して動作している既存設定** の現行値全量を事前ヒアリング項目に含めること。例: CloudFront の Custom Error Responses を変更するなら `DefaultRootObject`, `DefaultCacheBehavior`, `Origins`, `PriceClass` 等も JSON ダンプで保存。変更項目にフォーカスした確認だと、周辺設定の「偶然の正解」で成立していた挙動が崩れた時に後工程で発覚する（PRTP-752-fix-spa B チケット実装 retrospective P-2 より。`DefaultRootObject` 未設定が A チケット時点で見落とされ、B デプロイ時にルート URL が spa-shell に落ちる現象で顕在化）

## spec-writer
要件定義をもとに、実装可能な詳細仕様を作成するエージェントです。
### 呼び出しルール
- **要件定義完了後**：`@requirement-definer`の完了後に自動的に呼び出されます。
- **単独呼び出し**：既に要件が明確な場合は、直接`@spec-writer`を呼び出すことも可能です。
- **対応フロー**：
  1. 要件定義ドキュメントを読み込み
  2. 既存コードベースを調査
  3. 仕様ドキュメント（spec.md）を作成
  4. ユーザー承認後、自動的に`@task-writer`を呼び出し
- **起動プロンプトで明示すべき観点（重要）**：
  - 「エージェント定義の出力テンプレートに厳密に従え」（proken-typing-web prerendering より既出）
  - 「**フレームワーク固有のランタイム挙動**（Vue hydration / mount、React strict mode、Laravel middleware 順序、SSR→SPA マウント時の DOM 差し替え範囲等）で副作用が起きないかを設計段階で検討し、該当箇所は spec.md に明示すること。**既存 CSS/JS アセットを保持する生成物**（spa-shell.html 等）では、保持アセットが新規注入コンテンツ（スピナー等）に干渉しないかも検討すること（例: 既存の `#app` レイアウトスタイルが body flex 中央寄せを上書きして配置が崩れる等）」（PRTP-752-fix-spa 設計 retrospective + 実装 retrospective P-4 より）
  - 「**UI 要素の具体値**（色/サイズ/アニメーション速度等）を仕様に含める場合、『将来の機能追加・運用変更で変更可能性があるか』を設計者自身で問い、可能性があれば spec.md に **『将来変更候補』注記** を付けること。特に別チケットで関連スコープ変更が予定されている場合（例: SSG スコープ縮小と spa-shell の色分岐）は関連チケット ID を明記する」（PRTP-752-fix-spa 実装 retrospective より。B チケットで spa-shell 経由パスが変わる連鎖を設計時に想定できなかった）
  - 「**既存実装を変更する延長チケット（B チケット等）では、"変更前" のベースラインは実装済みコード（コミット済み）であり、前チケットの spec.md のサンプル実装ではない**。レビュー反映等で spec.md とコードが乖離している場合があるため、変更対象の関数/セクションについて実装済みコードをベースに記述すること」（PRTP-752-fix-spa B チケット設計 retrospective より。assertAssetIntegrity のサンプルが spec.md ベースで書かれ、実装済みコードの FORBIDDEN_IN_SHELL チェック等が欠落した）
  - 「**インラインスクリプトが DOM 要素（`document.body`, `document.querySelector` 等）にアクセスする場合、注入位置でその要素がパース済みであることを確認すること**。`</head>` 直前のスクリプトは `<body>` 内の要素にアクセスできない。DOM 操作が必要なスクリプトは `</body>` 直前に配置する」（PRTP-752-fix-spa B チケット設計 retrospective より。spec-writer が `</head>` 直前にスクリプトを配置し、`document.body` にアクセスできない設計を出力した）
  - 「**空 HTML / 派生生成物を設計する際は、既存 CSS のルート系セレクタを事前 grep 調査すること**。`spa-shell.html` のように『既存 `<link rel=stylesheet>` を保持したまま中身を空化する生成物』を設計する場合、既存 CSS ファイル（`dist/**/*.css` 等）で `html`, `body`, `#app`, `#root`, `.wrapper`, `.p-wrapper`, `main` 等のルート系セレクタに背景色/レイアウト指定が存在していないか awk/grep で全量抽出すること。存在する場合、ID 特異度（`#app` 等）が inline `<style>` の要素特異度（`body`）に勝ってしまうため、特異度競合の対策（`!important` or 子セレクタ）を spec.md に明記する」（PRTP-752-fix-spa B チケット実装 retrospective P-1 より。`app-style.css` の `html,body,#app,.wrapper{background-color:#f1f3f7}` が inline `<style>` の body 背景画像を隠蔽し、stg デプロイ後に「画像半分しか映らない」で発覚）
  - 「**既存実装の改修案件では、改修対象メソッドが他のメソッド・ファイルから参照されている場合、または改修対象が共有データ（saml_setting / 認可情報 / セッション情報 / 動的テーブル参照等）を扱う場合、その共有データを介して別メソッドの挙動が変わる経路を spec.md に網羅的に列挙すること**。特に **同 Controller / 同 Model 内の他メソッド** への波及は見落とされやすい。`grep -n "対象メソッド名\|共有データのカラム名" path/to/file.php` で能動的に確認し、確認結果を spec.md の『波及範囲』『副作用検討』セクションに明記する。並列レビューで spec.md 単体レビューだけしても別ファイル / 別メソッドの波及は検出困難であり、impact-analyzer に頼り切ると後工程で『要修正』指摘が出てスコープ拡大判断を強いられる」（cbt-p1-server CBT-684 設計 retrospective P-2 より。spec.md は `getAuthOrg::case 'admin'` 改修にフォーカスし、同 Controller の `postUpdateSamlSettings` の `User::resetPasswordForOrg` 経路の波及を見落とした。同 retrospective 過去 P-3 と類似 = 2 セッション連続の再発、構造的に潰す必要あり）
  - 「**Exception / Error クラスの throw 文を spec.md に焼き込む際は、対象クラスのコンストラクタ実装を Read で確認してから記述すること**。引数の意味（特に複数引数を取る場合の各引数の役割と整形仕様）を把握した上で書き、整形後のメッセージ実例（実際にユーザー / API クライアントが見る文字列）を spec.md に併記する。設計レビュー段階で『整形後メッセージが意図と整合しているか』を検証可能にする。Exception クラスの実装を確認せずサンプル実装を焼き込むと、整形ロジックに乗って不自然なメッセージになるケースが頻発する」（cbt-p1-server CBT-684 実装 retrospective P-2 より。`throw new ForbiddenException('write', 'admin AOG cannot be deleted (CBT-684)')` を焼き込んだ結果、`$target` が string の場合の整形仕様により `Cannot write this resource. 'admin AOG cannot be deleted (CBT-684)' privilege is required.` という不自然な文に。並列レビュー 3 種でも検出されず、実機検証で発覚し再修正）
  - 「**サーバー API のレスポンス種別（HTTP ステータス / エラーメッセージ / 新フィールド等）が変わる改修では、フロント側でその変化を UI に反映する必要があるかを設計段階で必ず判定すること**。判定の事前チェックとして、(1) フロントのグローバルエラーハンドリング（axios interceptor / loaderMixin / ApiAdminErrorHandler 等）が新レスポンスを扱えるか、(2) サーバー側翻訳基盤（lang/ja/messages.php 等）の整備状況、(3) フロント側のエラー表示パターン（alert / inline / toast / disabled 化等）を spec.md『§ フロント連携要否』セクションで明示する。要連携と判定された場合は MoSCoW Could 以上に『フロント側 UX 改修』タスクを必ず入れる。判定不能な場合は事前タスクで『フロント側との連携要否確認』を Phase 1 に積む」（cbt-p1-server CBT-684 実装 retrospective P-3/P-4 より。サーバー側 M-6 ガード追加だけで設計クローズした結果、画面で削除ボタン押下時に「クリックしても何も起きない」UX 不整合が実装段階で発覚。cbt-p1-client への急遽スコープ拡張 + 翻訳基盤の整備状況確認が後手に回って判断時間を浪費）
  - 「**AWS パイプライン系・自動化系の設計では『手動配置・EC2 への初期セットアップに依存する設計』を出力する前に、自動化経路を必ず検討すること**。検討対象: (1) CodeDeploy の組み込み環境変数（`APPLICATION_NAME` / `DEPLOYMENT_ID` / `DEPLOYMENT_GROUP_NAME` / `LIFECYCLE_EVENT` 等）, (2) CodeDeploy アーティファクト同梱（`appspec.yml` の `files` セクション）, (3) EC2 IMDS タグ取得, (4) SSM Parameter Store。これらで解決できない場合のみ手動配置を採用し、spec.md 内で『なぜ自動化困難か』の正当化を明示する。安易に `/sprix/etc/xxx` 等の手動配置ファイルを設計に組み込むとパイプライン化の趣旨と矛盾し、ユーザー指摘で書き直しになる」（keycloak-test INFO-57 retrospective P-1 より。`/sprix/etc/keycloak-cache-bucket` を EC2 に手動配置する設計を出力 → ユーザー指摘で `APPLICATION_NAME` 抽出方式へ書き換え）
  - 「**ハッシュ・署名・外部 API 提供形式の検証強化を仕様化する際は、対象リソースの提供形式を gh CLI / curl で実機確認すること**。security-reviewer 等から『SHA256 検証を Must に』のような指摘を受けた場合でも、推測で焼き込まず実機確認で提供形式を確定する。例: GitHub Releases は SHA256 を提供せず SHA1/MD5/GPG のみのプロジェクトもある。提供形式が想定と異なる場合は仕様修正前にユーザーに代替案（SHA1 中間案、GPG 署名検証等）を提示する」（keycloak-test INFO-57 retrospective P-3 より。SHA256 Must 格上げを仕様反映しかけたが、実機確認で Keycloak は SHA256 未提供と判明 → SHA1 中間案へ変更）

## task-writer
仕様書をもとに、実装タスクを分解・整理し、実行可能なタスク一覧を作成するエージェントです。
### 呼び出しルール
- **設計レビュー完了後**：`@risk-analyzer` + `@code-reviewer`（設計レビュー）の並列実行完了後に呼び出されます。
- **単独呼び出し**：既に仕様が明確な場合は、直接`@task-writer`を呼び出すことも可能です。
- **対応フロー**：
  1. 仕様ドキュメントを読み込み
  2. タスクを分解・整理
  3. タスク一覧ドキュメント（tasks.md）を作成
  4. ユーザー承認後、自動的に`@code-reviewer`を呼び出し
- **起動プロンプトで明示すべき観点（テスト系タスクの運用実態整合）**：
  - 「**ユニットテスト / 統合テスト / E2E テストの新規作成タスクを tasks.md に含める前に、プロジェクトの運用実態を確認すること**。具体的には `git log --since='1 year' --name-only -- tests/` 等で直近 1 年のテスト追加実績を調べ、実績が乏しい（例: 追加コミット 0〜2 件）場合は、(a) タスクを『スキップ』として代替検証方針（tinker 直叩き / UI 目視 / DB 観察等）を明記する、(b) 新規追加するとしても最小件数に絞る、のいずれかを選択する。仕様駆動で機械的にテスト作成タスクを入れるとプロジェクト実運用と乖離し、着手時に判断コストが発生する」（CBT-874 実装 retrospective P-3 より。直近 1 年テスト追加 0 件のプロジェクトで T-003「ユニットテスト作成」を入れてしまい、着手直前に運用整合性を相談する手戻りが発生）
  - 「**タスク粒度・順序制約・選択肢併記の禁止**: (1) **粒度**: spec.md §影響範囲 1 ファイルあたり最大 3 タスクを目安にし、それ以上のサブタスクは DoD の詳細化で吸収する（過度な分割でタスク数が肥大化すると後段の Edit で対象特定コストが乗る）。(2) **順序制約の埋め込み**: AWS パイプライン系等で『BeforeInstall → Install → AfterInstall』『CodeBuild 各フェーズは別シェルプロセスで export 引き継ぎなし』のような順序制約は spec.md から再解釈せず、tasks.md 内では一意の手順に絞り、選択肢併記（『A または B からの読込』等）は禁止。(3) **同時改修事項の明示**: 既存 typo / 既存 bug / shebang ミス等を本チケットで同時修正する場合は、各タスクの『詳細手順』に明示的に記載する」（keycloak-test INFO-57 retrospective P-3/P-4 より。tasks.md が 1697 行 / 85 タスクに肥大、T-207 で `.keycloak-version` 読取パスを `/sprix/etc/...` と revisionDir で選択肢併記したが前者はライフサイクル順序的に動作不可 → impact-analyzer IMP-001 として検出）

## security-reviewer
API実装コードおよびインフラ設定のセキュリティ脆弱性を専門的にチェックし、修正案を提示するエージェントです。
### 呼び出しルール
- **仕様作成完了後**：`@spec-writer`の完了後、`@risk-analyzer`・`@code-reviewer`と並列で呼び出します。設計段階では**STRIDEチェック（ST-01〜06）のみ**実施。
- **実装完了時**：`@code-reviewer`の完了・修正反映後に呼び出します（直列）。実装段階では**STRIDE + OWASPチェック全項目**を実施。品質レビューの修正を反映した状態でセキュリティチェックを行い、観点のブレを防ぐ。
- **単独呼び出し**：セキュリティ重点レビューが必要な場合に直接呼び出すことも可能です。
- **code-reviewerとの棲み分け**：code-reviewerは品質・規約・設計の総合レビュー（セキュリティは観点の一つ）。security-reviewerはSTRIDE + OWASP Top 10を軸にしたセキュリティ特化の深掘りチェック。
- **起動プロンプトで明示すべき観点（ファイル生成系の実装時）**：
  - 「**生成物の静的検証アサートの多層化**を確認すること。具体的には『削除対象が確実に削除されているか』『注入対象が確実に注入されているか』を、件数差分だけでなく直接文字列アサートで検証しているかを観点に含める。`replace` 系操作の silent no-op（対象不在時に何も置換されず素通しされるケース）を CI 段階で確実に検知できる構造になっているかをチェックすること」（PRTP-752-fix-spa 実装 retrospective より。SR-003 で `<meta robots noindex>` 注入失敗の silent no-op リスクが判明）
- **対応フロー**：修正案を提示し、**ユーザー確認後**に適用（勝手に適用せず、ワンクッション置くこと）

## coder
タスク一覧に基づき、仕様通りにコードを実装するエージェントです。
### 呼び出しルール
- **設計レビュー完了後**：`@code-reviewer`の設計レビュー完了後、ユーザーが実装を希望する場合に呼び出します。
- **単独呼び出し**：既にタスク一覧が存在する場合は、直接`@coder`を呼び出すことも可能です。
- **対応フロー**：
  1. タスク一覧（tasks.md）を読み込み
  2. 各タスクを順次実装
  3. **タスクごとにユーザー承認を待つ**
  4. 全タスク完了後、`@code-reviewer` → `@security-reviewer` を直列で呼び出し（`@test-writer` は独立、ユーザー任意のタイミング）
- **注意**：ユーザーが仕様設計のみを目的としている場合は、自動呼び出しをスキップします。
- **起動プロンプトで明示すべき観点（ツール権限エラー時の返送品質）**：
  - 「**Write/Edit ツールで権限エラーが発生した場合、差分を本文で返してください。その際、PHP / JS / TS 等のコード中の `>` `<` `&` `"` は HTML エンティティ（`&gt;` / `&lt;` / `&amp;` / `&quot;`）に変換せず、生の文字でそのまま返すこと。** 親側で Edit 適用する際にデコード工数が発生し、見落とせば壊れたコードが commit される」（CBT-874 実装 retrospective P-1 より。T-002 で coder が差分を `&gt;` エスケープ込みで返送し、親側で手動デコード＋Edit する手戻りが発生。coder エージェント定義の frontmatter `tools:` 未指定問題と複合して発火）
- **coder 起動を省略してよいタスク（親直接実施可）**（PRTP-752-fix-spa 実装 retrospective より）：
  以下に該当するタスクは、エージェント呼び出しオーバーヘッド（1 回あたり 2-5 分）を避けるため親エージェントが直接実施してよい。
  - **spec.md のサンプル実装を完全コピーするだけ**のタスク（設計時点でサンプル実装が焼き込まれているケース）
  - **既存ファイルへの単純 diff 挿入**（挿入位置 + 挿入内容が spec.md / tasks.md に完全明記されているケース）
  - **sitemap / config 等への定型エントリ追加**（1〜数行、既存形式踏襲のみ）
  一方、以下のタスクは coder 経由を維持すること：
  - ロジック設計や既存コードとの整合調整が必要なタスク
  - エラーハンドリングの判断が仕様に明記されていないタスク
  - 既存実装パターンの踏襲先を選択判断する必要があるタスク

## doc-updater
コードの変更を検知し、仕様書を逆更新するエージェントです。
### 呼び出しルール
- **実装完了後**：仕様書との整合性を確認し、差分があれば更新提案します。
