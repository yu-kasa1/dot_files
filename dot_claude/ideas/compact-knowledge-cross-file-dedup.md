# compact-knowledge に他ファイル横断重複検出を追加

> compact-knowledge スキルの検出軸を、現状の「ルール内部の重複」だけでなく**「他ファイル横断の重複」**まで広げる案。
> 発端: 2026-06-18 の `rules/agent-collaboration.md` ダイエット（366→266 行、約27%圧縮）。後半の各エージェント節（## db-analyzer 〜 ## doc-updater）が `~/.claude/agents/*.md` と内容重複していた事例。明示マーカー（"agents/X.md の必須観点セクションに集約済み" 等）が文中にあるのに本文も再掲されていた。

## 現状の compact-knowledge

`SKILL.md` 記載:
- 対象ファイル: `GLOBAL_MEMORY.md` / `pitfalls.md` / `CLAUDE.md` / `agents/*.md` / `rules/*.md`
- 検出軸: ルールの重複・未参照アンカー・統合候補
- 修正は実施せず候補レポートのみ出力

→ 「ルール内部の重複」軸はあるが、**ファイル横断で「同じ情報が複数箇所に再掲されている」**ケースは検出弱い（実例: 今回の agent-collaboration.md ↔ agents/*.md）。

## 追加したい検出軸

### 軸1: 集約済みマーカーからの逆引き

「リンクで足りる本文の重複」を機械検出するパターン。

検出ロジック:
1. 各ファイルから明示マーカーを grep:
   - `(集約済み|参照|詳細は.*\.md)`
   - `~/.claude/agents/.*\.md`
   - `rules/.*\.md (参照|を参照)`
2. マーカーを含むセクションを抽出
3. **同セクションに本文（リストや段落）が一定量（例: 3 行以上）存在**するなら「リンクで足りる候補」として列挙
4. 参照先ファイルに同等の内容が存在するか軽くチェック（任意、精度向上用）

### 軸2: キーワード3ファイル以上出現

「同じ概念が分散している」ケースの検出。

検出ロジック:
1. 強キーワード（エージェント名、技術用語、固有プロセス名）を辞書化
2. 各キーワードが何ファイルに出現しているかカウント
3. 3 ファイル以上に出ているものは「分散候補」として列挙（統合か単独配置か判断する材料）

## 期待効果

- rules/CLAUDE.md/agents/*.md のメンテで「太ってる感」を機械的に検出できる
- ダイエット作業を毎回手動 grep + 目視確認から、候補レポート起点に変えられる
- ドキュメント横断の整合性（同一概念が複数箇所で食い違って書かれている問題）も副次的に拾える

## ノイズ懸念

精度を上げると false positive が増える:
- "集約済み" と書いていても本文は補足説明（重複でない）のケース
- マーカー位置と本文がセクション境界をまたぐケース
- 同じキーワードでも文脈が違うケース（"spec.md" が単なる固有名詞 vs 仕様書フォーマット説明）

→ 検出結果はあくまで**候補レポート**にとどめ、確定削除は人間判定（compact-knowledge-apply に乗せる）。現状の compact-knowledge と同じ運用ポリシーで吸収可能。

## 実装メモ（叩き台、未検証）

- 1 軸目はシェルで grep + awk 程度で初版いけそう（過剰実装しない）
- 2 軸目はキーワード辞書のメンテが負荷になりがち。**まず軸1だけで様子見**、効果が出たら軸2を足すのが順当
- compact-knowledge-apply 側で「リンクで足りる候補を削除する変換」を半自動化できれば、ダイエット作業のサイクル時間が大きく縮む
