# 不具合調査統括エージェント（企画案）

## 背景
2026-05-26、PRTP-804 の不具合調査で手間取った振り返りから派生。

CLAUDE.md には PRTP-804 由来の「断定する前の1段検証」「同一仮説の評価が 3 回以上揺れたら机上の限界」「机上推測段階で強い表現を使わない」等のルールが既に積まれているが、ルールがあっても守れなかった/カバーしきれなかった部分が残った。

## 痛点（PRTP-804 ふりかえり）

調査の本質的な難しさは 2 層に分かれる:

1. **論点管理**: 本筋調査中に副次的な「爆弾候補」が次々浮上する。直接原因とは言い難いが無視もできない。スコープ判断が難しい
2. **爆弾の発火経路調査の信頼性**: 「網羅的に見た」と言いつつ周辺連携の見落としで不発判定 → 後から新論点が浮上、を繰り返した

### 既存エージェントとの差分
- `impact-analyzer` は「変更予定の影響範囲」を見る**前向き**ツール。「既存の潜在バグの発火経路を網羅する」**後ろ向き**調査とは観点が違う
- `Explore` は read-only 検索特化。「論点の管理」「仮説の更新」「網羅性の担保」までは責務外
- 現状、調査専用の枠組みはない

## 設計方針（雑案）

調査は「情報を絞る」より「情報を集めて統合する」作業。並列化と情報量増加が噛み合う領域なので、**統括エージェントの元で動的に並列 spawn する構造**が合うと考える。

レビュー系の並列（spec-writer 後の risk / code / security 3並列）は**静的並列**だが、調査は**動的並列** —— 調査中に論点 B が浮上したら即サブ調査を spawn —— のほうが合う。

### 構成案
- **investigation-coordinator**（統括役）
  - 報告文脈 / 仕様前提の分離宣言（CLAUDE.md 既存ルール）
  - 論点管理（本筋 / 別チケット候補 / 無視可 の仕分け）
  - 強表現抑止（机上推測段階で「主犯確定」「最有力」等を使わせない）
  - 仮説変動カウンタ（3 回揺れたら机上の限界、実機検証ピボット）
  - サブ spawn 判断と結果統合

- **investigator サブエージェント**（並列 spawn される側）
  - 論点ごとに 1 つずつ spawn
  - 「この論点の発火経路を網羅」「不発なら不発と明示」を返す
  - 新論点が見つかったら統括に報告（統括が追加 spawn 判断）

### トレードオフ
- サブ間でコンテキスト共有できない → 論点 A 調査の知見が論点 B に活きない
  - 緩和策: 統括が結果集約 → 次回 spawn 時にプロンプトへ織り込む
- 並列起動オーバーヘッド（1 サブ 2-5 分） vs 直列調査の時間
  - 大規模調査（論点 5+ 件）では並列が効くはず。小規模調査では統括だけで十分

## エージェント vs スキル

会話コンテキスト依存度が高い作業（retrospective 等）はスキル化のほうが向くが、本件は **並列 spawn が要の設計** のためエージェントが妥当。統括自体は会話コンテキストで動く（メインエージェントが統括役を担う形でもよい）。

整理:
- 統括 = メインエージェント or `investigation-coordinator` という独立エージェント
- サブ = `investigator` エージェント（複数並列）

## 想定起動シナリオ
- Backlog 課題 + 関連リポジトリ群を入力に「不具合調査を開始」と発話
- 統括が初期論点を 1-3 件抽出 → investigator を並列 spawn
- 各サブが発火経路調査結果を返す（不発 / 発火可能 / 新論点発見）
- 統括が結果統合 → ユーザーへ中間報告 → 追加 spawn 判断

## 試運転計画

「不具合調査はやろうと思って湧いてくるもんでもない」（事例待ちでは要件固まらない）ため、idea まとめた段階で試作着手して実戦投入してダメなら直す方針。

### 試作ステップ
1. `investigation-coordinator` と `investigator` の最小実装（YAML フロントマター + システムプロンプト骨格）
2. 既存の PRTP-804 振り返り内容をシナリオとして食わせて、机上動作確認
3. 次回の不具合調査タスクで実戦投入

### 評価ポイント
- 論点管理が正しく機能するか（本筋 / 別チケット候補 / 無視可 の仕分け精度）
- 並列 spawn のオーバーヘッドが調査全体の時短に見合うか
- サブ間のコンテキスト分断が痛いか（統括の織り込みで緩和できるか）

## 2026-05-26 詰めメモ

### Explore との棲み分け（investigator = Explore の強化版と位置付け）

| | Explore | investigator |
|---|---|---|
| 読み方 | excerpts | 全文 Read |
| 目的 | 場所特定 | 発火経路の判定 |
| 出力 | ファイルパス / 行 | 不発 / 発火可能 / 新論点 |
| 網羅性 | 不問 | 担保必須 |
| 強表現抑止 | 不問 | 必須（机上推測段階） |

Explore が「軽量・場所特定」に振り切ってる設計の逆方向（重い・判定責任引き受け）で investigator を作る。Explore を孫として呼ぶ二段構成は情報劣化が怖いので、investigator が直接 Glob/Grep/Read 使う想定。

### 痛点 3 種と対策（PRTP-804 ふりかえり）

#### ① 発火経路の見落とし（情報不足）
- 起動時の**強制走査**: 対象シンボルの「使う側」「使われる側」両方向 grep を必須化
- **「読まなかったファイル」リスト**を明示返却（暗黙の見落としを可視化）
- Explore 的「該当箇所候補列挙」を**サブの最初の工程**として焼き込む

#### ② 特異性度外視ストーリーの乱造（情報過剰）
新論点と既存論点の関連付けで「机上の理論上可能」レベルのレアシナリオを発火可能扱いして人力フィルタを強いる問題。CLAUDE.md「通常運用で到達可能なシナリオを1つ以上書き出せるか」原則を出力フォーマットに焼き込む:

- シナリオごとに**発火条件 / 通常運用での到達頻度 / 机上のみフラグ**を必須出力
- 「机上の理論上のみ」ラベルが付いたものは統括が自動で『別チケット候補 / 無視可』へ振り分け
- 出力自体は禁じない（仕分けで吸収する）

#### ③ 不具合をカバーする記述の見落とし → 不発ストーリーの乱造
CLAUDE.md「Validator/Rule/Filter 単体修正前のチェーン確認」の**調査版**。発火可能判定の前に防御コード探索を必須化:

- **前段ガード**: バリデーション / FormRequest / middleware / Guard / Policy
- **フレームワーク内部処理**: Laravel `Handler::report()`/`render()`、自動エスケープ等（[#info-126-p1-p2] 同根）
- **後段の正規化/エスケープ**: htmlspecialchars / SQL バインディング等
- **同関数内の既存チェック**: early-return / try-catch
- いずれかでカバーされていれば**不発**判定
- **「防御コード未探索のまま発火可能判定は禁止」** をシステムプロンプトに焼き込む

### 出力フォーマット案

```
シナリオ: XXX
発火条件: ...
通常運用での到達頻度: 日常 / 稀 / 机上のみ
防御コード探索結果:
  - 前段ガード: 探索済み → なし / あり（場所: ...） / 未探索
  - フレームワーク内部処理: ...
  - 後段正規化: ...
  - 同関数内既存チェック: ...
読まなかったファイル: [...]（網羅性の可視化）
判定: 発火可能 / 不発 / 要追加調査
```

ルール:
- 「未探索」がある状態で「発火可能」判定は出せない
- 「机上のみ」ラベルが付いたシナリオは統括が別チケット候補/無視可 に自動振り分け

### 統括の設計（2026-05-26 詰め）

**統括は独立エージェント（`investigation-coordinator`）で立てる**。理由:
- システムプロンプトに PRTP-804 由来のルール群を焼き込める（1段検証、強表現抑止、仮説変動カウンタ、防御コード探索必須）
- メインエージェントが調査管理から解放される → ユーザーとの会話を維持できる
- 「投げて待つ」型の運用に向く

#### 統括の自律範囲
- サブ spawn 判断（並列数、粒度）
- 結果統合
- 論点の本筋 / 別チケット候補 / 無視可 仕分け
- 「机上のみ」ラベル付きシナリオの自動振り分け

#### 人間に戻す 4 チェックポイント

| # | 契機 | 性質 | 動作 |
|---|---|---|---|
| ① | 初期論点まとめ時点 | 方向性合意 | 必ず停止して待つ |
| ② | 初期仮説が全部不発 | フレーミング失敗の疑い | 停止してピボット相談 |
| ③ | 新論点が初期論点超え | スコープ膨張 | 通知して継続 or 停止判断 |
| ④ | 同一仮説の評価変動 3 回 | 机上深掘りの限界サイン | 停止 → 実機検証ピボット相談 |

①でユーザー OK した論点リスト = 初期論点として確定。以降は新論点として境界が明確化する。

#### 「広く見る」一次対策（揺れを未然に防ぐ）

④は安全網。本来は揺れないように設計するのが筋。①〜③の対策とほぼ重複するが、「広く見る」設計が機能していれば揺れも減る:

- 起動時の両方向 grep 強制（対象シンボルの「使う側」「使われる側」）
- 読まなかったファイルの明示返却（暗黙見落としの可視化）
- 防御コード探索を発火判定前に必須化（後から「実はガードあった」で反転するのを防ぐ）
- 並列 spawn で論点を一気に広げる（後出し論点で評価がひっくり返る頻度を下げる）

**位置付け**: 「広く見る」設計 = 一次対策（揺れを未然に防ぐ） / 4 チェックポイント = 二次対策（限界検知）

#### 最終判断は必ず人間
- 「主犯確定」等の強表現アウトプット
- 別チケット起票
- 修正方針

### サブの粒度（2026-05-26 詰め）

**ハイブリッド方式**: 統括が論点ごとに「明らかに独立」or「相互に絡む可能性あり」を判断して spawn 粒度を決める。

- 第一波は独立 spawn（1論点1サブ）
- サブの返答で**「他論点との関連可能性」フラグを必須返却**
- 統括が結果を見て第二波で束ねる判断ができる構造

PRTP-804 で起きた「独立に見えた論点が実は同じ原因」パターンへの備え。

### サブと統括の役割分担（2026-05-26 詰め）

**サブ = 材料収集 + 仮判定 / 統括 = 統合最終判定**。

#### サブ（investigator）の出力ルール
- 判定（発火可能 / 不発 / 要追加調査）+ **根拠** + **不確実性** を必ず返す
- 強表現禁止: 「主犯確定」「最有力」「サイレントデータ破壊バグ」等は使わない（CLAUDE.md [#prtp-804-investigate-p1] 継承）
- 「未探索」が残ってる状態で「発火可能」判定は出せない
- **「サブの判断で材料を間引かない」**: 不確実性が高くても、稀シナリオでも、判定は出す（机上のみフラグは付けてよいが除外しない）
- 防御コード探索結果は「あり / なし / 未探索」を必ず返す（見つからなくても探した事実は返す）
- 読まなかったファイルリスト必須返却

#### 統括（coordinator）の役目
- サブ判定は**仮判定**として扱う
- 全サブの統合視点 + 仮説変動カウンタ + 「机上のみ」フィルタを通して**最終判定**
- 「未探索」が残ってる仮判定は自動で「要追加調査」に格下げ → 再 spawn
- 選別は統括の専権、サブは収集に徹する

## 試運転記録

### 2026-05-27 第1回試運転
- **題材**: info-assist-pack / info-practice の laravel.log に出る KeycloakCallbackError（OAuth コールバック処理での例外、message 空、AuthService.php:59 起点）
- **ゴール**: 発生手順の明確化（原因特定・修正は次フェーズ）
- **動作感**: agent team の動き（並列 spawn / チェックポイント / レポート上がり）は理想通りに機能。「投げて待ってればレポートが上がってくる」運用が成立
- **限界**: 題材に根本っぽい原因がなく、結論は推測止まり。これは agent team の限界ではなく題材側の都合
- **試運転で追加されたルール**:
  - `#info-keycloak-investigate-p1`: 初期論点抽出時に**症状前提（頻度・間隔等）を疑う論点を 1 つ含める**。前提を所与にすると論点設定全体が引きずられる
  - `#info-keycloak-investigate-p2`: Critical/High ラベル付与前に**成立条件すべてが机上確認済みか必須チェック**。未確認は「(条件確認中)」サフィックスを付ける
  - 両ルールは agent 定義（手順 2・手順 7）に反映済み、pitfalls.md にもアンカー追加済み

## 将来の発展方向: 汎用ターミナル化

不具合調査専用に留めず、agent team の枠組み（コーディネータ + 動的並列 spawn + チェックポイント + 仕分け）を他用途にも転用したい。

### 想定転用先
- **レガシーコード考古学**: 「なぜこの実装こうなってるのか」をファイル / コミット履歴 / 周辺仕様横断で調査
- **パフォーマンス調査**: ボトルネック候補が複数浮かぶケース、並列で経路追跡
- **大規模リファクタリングの影響サーベイ**: impact-analyzer の動的並列版（複数モジュール跨ぐ）
- **設計探索の前段**: feasibility-writer より広い「問題空間自体の把握」フェーズ
- **コードベース全体の品質サーベイ**: 特定観点（N+1 全箇所、未処理例外全箇所等）で網羅的に洗う

共通点: 「ゴールはあるが論点が動的、網羅性が要る、人間が途中介入したい」。

### 構造案: 統括は汎用ターミナル、サブはドメイン特化
- **統括（汎用）**: チェックポイント / 仮説変動カウンタ / 仕分けロジックは普遍 → 汎用化（`research-coordinator` 的なリネーム候補）
- **サブ（specialist）**: ドメイン固有のディシプリンを各サブが持つ
  - `investigator`（不具合調査）= 強表現禁止 / 報告文脈分離 / 防御コード探索
  - `legacy-archeologist`（レガシー考古学）= git blame / 当時の commit message / PR 議論サルベージ
  - `perf-investigator`（パフォーマンス調査）= profiler 出力 / SQL EXPLAIN / N+1 経路追跡
  - 等々
- 統括は呼び出し時にどの specialist を spawn するか指示を受ける or 入力から判断

新ドメイン追加はサブ 1 個追加で済む構造。統括を量産しなくていい。

### 着手判断
- 具体的なリネーム / 構造変更は **2 個目の specialist を試作するタイミング**で判断
- 早すぎる抽象化は痛い目見がちなので、しばらく `investigation-coordinator` を使い込む期間を置く
- 2 個目の用途が湧いてきたら、まず単純コピーで `xxx-coordinator` を作るのではなく、共通項抽出から始める
